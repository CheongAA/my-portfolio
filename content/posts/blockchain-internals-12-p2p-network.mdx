MetaMask에서 "전송" 버튼을 누르는 순간, 내 트랜잭션은 어떻게 전 세계 이더리움 노드에 전달될까?

중앙 서버에 보내는 게 아니다. 이더리움에는 "본부 서버"가 없다. 트랜잭션은 P2P(Peer-to-Peer) 네트워크를 통해 퍼진다. 이 구조가 어떻게 동작하는지 살펴보자.

---

## 이더리움의 P2P 프로토콜: devp2p

이더리움 실행 레이어 노드들은 **devp2p**라는 프로토콜로 통신한다. 그 위에 두 가지 서브프로토콜이 있다.

**RLPx**: 노드 간 암호화된 TCP 연결을 맺는 전송 레이어.

**ETH protocol (eth/68)**: 블록, 트랜잭션, 상태 데이터를 교환하는 응용 레이어.

합의 레이어(비콘 체인)는 별도로 **libp2p**를 사용한다. 실행 레이어와 합의 레이어가 다른 P2P 프로토콜을 쓰는 이유는 역사적으로 따로 개발됐기 때문이다.

---

## 피어 탐색: 어떻게 다른 노드를 찾나

인터넷 어딘가에 있는 이더리움 노드들을 어떻게 찾을 수 있을까?

이더리움은 **discv5(Discovery Protocol v5)**를 쓴다. DHT(Distributed Hash Table) 기반의 분산 피어 탐색 프로토콜이다.

처음 노드를 시작할 때는 하드코딩된 **부트스트랩 노드** 목록을 활용한다. 이더리움 재단이 운영하는 공신력 있는 노드들이다.

```
enode://d860a01f9722d78051619d1e2351aba3f43f943f6f00718d1b9baa4101932a1f5011f16bb2b1bb35db20d6fe28fa0bf09636d26a87d31de9ec6203eeedb1f666@18.138.108.67:30303
```

부트스트랩 노드에 연결되면 "주변 노드 목록을 알려줘"라고 요청하고, 거기서 또 다른 노드를 찾는다. 카카오 친구 찾기처럼 연쇄적으로 피어가 늘어난다.

---

## 트랜잭션 전파: 가십 프로토콜

피어와 연결된 후, 트랜잭션은 **가십(Gossip)** 방식으로 전파된다.

```
나 → 피어 A, B, C에게 트랜잭션 전달
피어 A → 자신의 피어 D, E, F에게 전달
피어 B → 자신의 피어 G, H에게 전달
...
```

소문이 퍼지는 것처럼 네트워크 전체로 빠르게 확산된다.

같은 트랜잭션을 두 번 전달받아도 mempool에 이미 있으면 무시한다. 무한 루프가 생기지 않도록.

이더리움은 두 단계로 트랜잭션을 전파한다.

1. **NewPooledTransactionHashes**: 먼저 트랜잭션 해시만 전파한다.
2. 상대 노드가 "이 해시 모르는데, 내용 줘"라고 요청하면 전체 트랜잭션 데이터를 전송한다.

이렇게 하면 이미 아는 트랜잭션은 다시 받지 않아도 돼서 네트워크 대역폭을 절약한다.

## 노드 유형: 이더리움 네트워크의 참여자들

모든 노드가 똑같이 동작하는 건 아니다.

### 풀 노드 (Full Node)
블록 헤더와 최근 상태를 저장한다. 새 블록의 모든 트랜잭션을 직접 검증한다. 네트워크의 뼈대. 이더리움 실행 클라이언트(Geth 등)가 기본적으로 이 모드로 동작한다.

### 아카이브 노드 (Archive Node)
풀 노드에 더해 모든 과거 상태를 저장한다. "블록 #5,000,000 시점의 특정 주소 잔액은?" 같은 쿼리를 답할 수 있다. 저장 공간이 수 TB 이상 필요하다. Etherscan 같은 블록 익스플로러가 아카이브 노드를 운영한다.

### 라이트 노드 (Light Node)
블록 헤더만 저장한다. 트랜잭션 검증은 풀 노드에 의존해서 Merkle Proof로 확인한다. 저장 공간이 작아서 스마트폰이나 IoT 기기에 적합하다.

## 동기화 방식

새 노드가 네트워크에 합류하면, 수백만 개 블록의 데이터를 가져와야 한다. 이걸 어떻게 효율적으로 할까?

### Full Sync (풀 동기화)
제네시스 블록(0번)부터 현재까지 모든 블록을 다운받아서 직접 실행한다. 가장 안전하지만 시간이 수일 걸린다.

### Snap Sync
현재 상태 트리(State Trie) 스냅샷을 먼저 다운받는다. 수백 GB의 과거 상태 대신 최신 상태만 받아서 빠르게 시작한다. 이후에 블록 헤더를 검증하면서 완전한 풀 노드가 된다. 현재 Geth의 기본 동기화 방식이다.

### Light Sync
블록 헤더만 다운받는다. 아주 빠르지만, 트랜잭션 검증을 위해 항상 풀 노드에 의존해야 한다.

## 현재 이더리움 네트워크 규모

2026년 기준:
- 실행 레이어 풀 노드: 약 6,000개 이상
- 합의 레이어 검증자: 약 100만 명 이상

노드가 특정 국가나 클라우드 제공자에 집중되는 것도 탈중앙화 리스크로 꼽힌다. 현재 상당수의 이더리움 노드가 AWS, Hetzner 같은 클라우드에서 운영된다. 이 클라우드 업체들이 서비스를 거부하면 네트워크에 영향이 생길 수 있다.
