2022년 9월 15일, 이더리움은 PoW에서 PoS로 전환했다. 전력 소모량이 99.95% 줄었고, 업계에서 가장 큰 기술적 전환 중 하나로 꼽힌다.

그런데 이 전환을 위해 이더리움은 독특한 아키텍처를 선택했다. 기존 체인을 통째로 교체하는 대신, **두 개의 레이어로 분리**했다.

---

## 왜 병렬로 실행했나

PoW에서 PoS로 전환하는 방법은 크게 두 가지다.

**방법 1**: 특정 블록 높이에서 합의 방식을 교체한다.

이건 위험하다. 만약 전환 코드에 버그가 있으면 이더리움 메인넷이 멈출 수 있다. 수십조 원 규모의 자산이 담긴 네트워크에서 이런 위험을 감수하기 어렵다.

**방법 2**: 새 PoS 체인을 먼저 만들어서 충분히 검증한 뒤, 기존 PoW 체인과 연결한다.

이더리움이 선택한 방법이다.

2020년 12월, **비콘 체인(Beacon Chain)**이 PoW 메인넷과 별개로 출시됐다. 약 2년간 메인넷 자산 없이 순수하게 합의 메커니즘만 테스트하는 체인으로 운영됐다.

2022년 9월, 이 두 체인이 합쳐졌다. 이게 **The Merge**다.

---

## 두 레이어의 역할

The Merge 이후 이더리움 노드는 두 개의 소프트웨어를 동시에 실행한다.

### 실행 레이어 (Execution Layer, EL)
- 트랜잭션 처리, EVM 실행, 상태 변경 담당
- 기존 PoW 시절 이더리움 클라이언트(geth 등)가 이 역할
- 대표 클라이언트: **Geth**, Besu, Nethermind, Erigon

### 합의 레이어 (Consensus Layer, CL)
- 비콘 체인. 검증자 관리, 블록 제안/투표, Finality 담당
- PoS 로직 전체가 여기에 있다
- 대표 클라이언트: **Lighthouse**, Prysm, Teku, Nimbus

```
합의 레이어 (Lighthouse)
│  - 다음 Proposer 결정
│  - 검증자 투표 수집
│  - Finality 처리
│
│  Engine API (HTTP)
│
실행 레이어 (Geth)
   - 트랜잭션 실행
   - 상태 변경 (stateRoot 업데이트)
   - mempool 관리
```

## Engine API: 두 레이어가 대화하는 방법

합의 레이어와 실행 레이어는 **Engine API**라는 HTTP 인터페이스로 통신한다.

주요 메서드 세 가지:

### engine_forkchoiceUpdated
합의 레이어가 실행 레이어에게 "현재 정당한 체인의 최신 블록이 이거야"라고 알려준다.

```json
{
  "headBlockHash": "0x...",    // 현재 체인 최신 블록
  "safeBlockHash":  "0x...",   // Justified된 블록
  "finalizedBlockHash": "0x..." // Finalized된 블록
}
```

### engine_getPayload
합의 레이어가 "다음 블록에 담을 트랜잭션 묶음 줘"라고 요청한다. 실행 레이어는 mempool에서 트랜잭션을 골라서 반환한다.

### engine_newPayload
합의 레이어가 새로 받은 블록을 실행 레이어에게 전달해서 "이거 실행해봐, 유효한지 확인해"라고 한다. 실행 레이어는 EVM으로 트랜잭션을 처리하고 결과를 반환한다.

## 실제 블록 생성 흐름

매 Slot(12초)마다 이런 일이 벌어진다.

```
1. [CL] RANDAO로 이번 Slot의 Proposer 결정

2. [CL → EL] engine_getPayload 호출
   "이번 블록에 담을 트랜잭션 목록 줘"

3. [EL] mempool에서 적절한 트랜잭션 선택, 블록 바디 구성
   → payload 반환

4. [CL] payload를 포함한 블록 서명 후 P2P 네트워크에 전파

5. [다른 노드의 CL] 블록 수신
   → [자신의 EL] engine_newPayload 호출로 실행 검증

6. [다른 노드의 CL] 유효하면 Attestation(투표) 전파

7. [모든 CL] 투표 집계 → LMD-GHOST로 최선 체인 결정
```

## 클라이언트 다양성이 왜 중요한가

이더리움에는 실행 레이어 클라이언트가 4종, 합의 레이어 클라이언트가 4종 있다.

왜 여러 개가 필요할까?

만약 검증자 80%가 같은 클라이언트를 쓰고 그 클라이언트에 버그가 있다면? 버그가 있는 블록에 80%의 검증자가 동의하게 된다. 이 경우 Casper FFG는 Safety를 위해 Finality를 멈추겠지만, 이미 잘못된 상태가 전파될 수 있다.

반면 어떤 클라이언트도 전체의 1/3을 넘지 않는다면, 한 클라이언트에 버그가 생겨도 나머지 2/3가 올바르게 동작하면서 Finality를 유지할 수 있다.

현재 실행 레이어에서 Geth 점유율이 지나치게 높다는 점이 이더리움 커뮤니티에서 꾸준히 우려되는 문제다.

## The Merge 이후 달라진 것들

| 항목 | The Merge 이전 | The Merge 이후 |
|------|-------------|-------------|
| 합의 방식 | PoW (채굴) | PoS (스테이킹) |
| 블록 생성 주기 | 평균 13초 (불규칙) | 정확히 12초 |
| ETH 발행량 | 연 4.3% | 연 0.5% 미만 |
| 전력 소모 | 연 77 TWh | 연 0.01 TWh |
| 클라이언트 수 | 실행만 | 실행 + 합의 |

블록 생성 주기가 정확해진 것도 흥미롭다. PoW는 채굴이 확률적 과정이라 블록 생성 시간이 들쭉날쭉했다. PoS는 Slot이 12초로 고정되어 있어서 훨씬 예측 가능해졌다.
