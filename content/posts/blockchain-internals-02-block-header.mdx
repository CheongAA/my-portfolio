지난 편에서 Merkle Tree를 통해 수많은 트랜잭션이 해시 하나(txRoot)로 요약된다는 걸 봤다. 그 txRoot는 어디에 저장될까? 바로 **블록 헤더**다.

블록 헤더는 블록 전체의 요약본이다. 수 메가바이트짜리 블록 데이터 중에서 핵심 메타데이터만 추려낸 수백 바이트의 구조체다. 이걸 뜯어보면 블록체인이 왜 "불변"인지 구조적으로 이해할 수 있다.

---

## 블록의 두 부분: 헤더와 바디

블록은 크게 두 부분으로 나뉜다.

```
블록
├── 헤더 (Header)  ← 수백 바이트, 메타데이터
└── 바디 (Body)    ← 수 MB, 실제 트랜잭션 목록
```

노드들이 블록 해시를 계산하거나 블록 유효성을 빠르게 확인할 때는 헤더만 쓴다. 트랜잭션 내용 전체를 매번 다 들여다볼 필요가 없다.

---

## 블록 헤더 필드 해부

이더리움 블록 헤더의 주요 필드를 하나씩 살펴본다. The Merge 이후(PoS 기준)다.

### parentHash
```
parentHash: 0x4a3f9c...
```

이전 블록의 해시값이다. 이게 블록들을 "체인"처럼 엮는 핵심이다.

만약 누군가 블록 100번의 트랜잭션을 수정하면? 블록 100번의 해시가 바뀐다. 블록 101번의 parentHash가 "예전 블록 100번 해시"를 가리키므로 일치하지 않는다. 블록 101번을 수정하면 블록 102번이 안 맞는다. 체인 끝까지 전부 다시 계산해야 한다. 수천 개 노드에서 동시에.

**parentHash 하나가 블록체인의 불변성 전체를 책임진다.**

### stateRoot
```
stateRoot: 0x2d8f7a...
```

이더리움에 존재하는 **모든 계정의 현재 상태**를 담은 Patricia Merkle Trie의 루트 해시다. 모든 지갑 잔액, 모든 컨트랙트 코드, 모든 컨트랙트 스토리지가 이 루트 하나에 요약돼 있다.

이 블록 이후 누군가의 잔액이 변하면 stateRoot도 바뀐다. 두 노드의 stateRoot가 같다면 전체 상태가 완전히 동일하다는 뜻이다.

### transactionsRoot (txRoot)
```
transactionsRoot: 0x9b4c1e...
```

이 블록에 담긴 트랜잭션 목록의 Merkle Root다. 지난 편에서 자세히 다뤘다. 이 값으로 특정 트랜잭션이 이 블록에 포함됐는지 Merkle Proof로 검증할 수 있다.

### receiptsRoot
```
receiptsRoot: 0xe7a2f0...
```

각 트랜잭션의 실행 결과(영수증)들의 Merkle Root다. 영수증에는 가스 사용량, 성공/실패 여부, 이벤트 로그 등이 담긴다.

"이 트랜잭션이 성공했나요?"를 영수증 없이도 Merkle Proof만으로 검증할 수 있는 이유가 이 필드 덕분이다.

### number
```
number: 19482031
```

블록 높이. 제네시스 블록이 0번이고 하나씩 증가한다. 이 숫자는 체인 길이를 나타낸다.

### timestamp
```
timestamp: 1741511123
```

블록이 생성된 시간 (Unix timestamp, 초 단위). PoS에서 슬롯 시간(12초)에 맞춰서 생성된다.

### gasLimit / gasUsed
```
gasLimit: 30000000
gasUsed:  12483920
```

`gasLimit`은 이 블록에서 소모할 수 있는 최대 가스량이다. `gasUsed`는 실제로 소모된 양이다. 이 비율로 다음 블록의 baseFee가 결정된다. gasUsed가 목표치(gasLimit의 50%)를 넘으면 baseFee가 오르고, 못 미치면 내려간다.

### baseFeePerGas
```
baseFeePerGas: 12000000000  (12 Gwei)
```

EIP-1559로 도입된 필드다. 이 블록에서 트랜잭션이 내야 하는 기본 수수료. 블록마다 자동 조정되고 **전액 소각**된다. 검증자에게 가는 게 아니라 ETH 공급량을 줄이는 데 쓰인다.

### feeRecipient
```
feeRecipient: 0xAbCd...
```

PoW 시절엔 `miner`라는 필드에 채굴 성공한 노드 주소가 들어갔다. PoS 전환 후엔 `feeRecipient`로 이름이 바뀌었다. 블록 제안자가 트랜잭션 수수료(priorityFee)를 받을 주소다.

### extraData
```
extraData: 0x...
```

최대 32바이트까지 임의 데이터를 넣을 수 있는 필드. 채굴 시절엔 채굴 풀이 여기에 서명을 넣었다. 현재는 클라이언트 버전 정보 등을 넣는 경우가 많다.

## PoW 시절엔 있었지만 지금은 없는 필드들

The Merge 이후 삭제되거나 의미가 없어진 필드들이 있다.

| 필드 | PoW 역할 | PoS 이후 |
|------|---------|---------|
| `difficulty` | 채굴 난이도 | 고정값 0 |
| `nonce` | 채굴 시 탐색하는 숫자 | 고정값 0x0000000000000000 |
| `mixHash` | PoW 증명값 | prevRandao(무작위성 제공)로 재활용 |

PoS에서 `mixHash`는 이름이 `prevRandao`로 바뀌어, 검증자들이 생성하는 무작위값을 담는다. 이 무작위값이 검증자 선출에 쓰인다. (4편에서 자세히)

## 블록 해시는 어떻게 계산되나

블록 해시는 헤더 필드 전체를 RLP 인코딩한 뒤 keccak256으로 해시한 값이다.

```
blockHash = keccak256(RLP(헤더 필드 전체))
```

RLP(Recursive Length Prefix)는 이더리움의 직렬화 방식이다. JSON.stringify처럼 데이터 구조를 바이트열로 변환하는 포맷인데, JSON보다 훨씬 간결하고 결정론적이다. 같은 데이터를 RLP로 인코딩하면 항상 같은 바이트열이 나온다.

## 실습: Etherscan에서 직접 확인해보기

Etherscan에서 아무 블록이나 골라서 헤더 값을 직접 볼 수 있다.

1. etherscan.io 접속
2. 아무 블록 번호나 입력
3. Block Height, Timestamp, Fee Recipient 확인
4. "More Details" 펼치면 Extra Data, Burnt Fees 등 확인 가능

stateRoot나 txRoot 같은 raw 헤더 필드는 `eth_getBlockByNumber` RPC 콜로 확인할 수 있다.

```bash
curl https://mainnet.infura.io/v3/YOUR_KEY \
  -X POST \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"eth_getBlockByNumber","params":["latest",false],"id":1}'
```

응답에 `stateRoot`, `transactionsRoot`, `receiptsRoot`, `parentHash` 등이 전부 나온다.
