AI API를 자주 쓰다 보면 비용이 쌓인다. 특히 같은 시스템 프롬프트나 문서를 매 요청마다 반복해서 보내는 경우가 많다. Prompt Caching은 이 비용을 크게 줄여준다.

---

## 문제: 같은 내용을 계속 보낸다

RAG 시스템을 만들었다고 하자. 매 질문마다 이런 구조로 요청을 보낸다:

```
시스템 프롬프트 (500 토큰)
+ 검색된 문서 컨텍스트 (3000 토큰)
+ 사용자 질문 (50 토큰)
= 총 3550 토큰
```

하루 1000번 요청하면 시스템 프롬프트 500,000 토큰 + 문서 3,000,000 토큰이 중복으로 소비된다. 질문 내용 50,000 토큰을 제외한 나머지 대부분이 반복이다.

---

## Prompt Caching이 뭔가

처음 요청 시 지정된 부분을 캐시에 저장한다. 이후 같은 내용이 포함된 요청이 오면 캐시에서 불러온다.

캐시에서 읽는 비용은 새로 처리하는 비용보다 훨씬 싸다.

Anthropic 기준:

- 캐시 쓰기: 일반 입력 비용 × 1.25
- 캐시 읽기: 일반 입력 비용 × 0.1 (**90% 절감**)

처음 한 번만 쓰기 비용을 내고, 이후 동일 내용은 10% 비용으로 처리된다.

---

## Anthropic에서 Prompt Caching

캐시할 부분에 `cache_control: {"type": "ephemeral"}` 을 붙인다.

```python
import anthropic

client = anthropic.Anthropic()

# 긴 시스템 프롬프트를 캐싱
response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": """
            너는 이 회사의 내부 문서 전문가야.

            [회사 제품 문서 전체 — 수천 줄]
            [API 레퍼런스 전체]
            [내부 가이드라인]
            """,
            "cache_control": {"type": "ephemeral"}  # 이 부분을 캐싱
        }
    ],
    messages=[
        {"role": "user", "content": "결제 API 사용 방법 알려줘"}
    ]
)

# 캐시 사용 현황 확인
usage = response.usage
print(f"캐시 생성: {usage.cache_creation_input_tokens} 토큰")
print(f"캐시 읽기: {usage.cache_read_input_tokens} 토큰")
print(f"일반 입력: {usage.input_tokens} 토큰")
```

두 번째 요청부터는 `cache_read_input_tokens`가 늘어나고 비용이 크게 줄어든다.

---

## 여러 부분에 캐싱 적용

여러 위치에 캐시를 설정할 수 있다. 캐시 포인트는 최대 4개.

```python
response = client.messages.create(
    model="claude-opus-4-6",
    system=[
        {
            "type": "text",
            "text": "기본 시스템 프롬프트 (변경 없음)",
            "cache_control": {"type": "ephemeral"}  # 캐시 포인트 1
        }
    ],
    messages=[
        {
            "role": "user",
            "content": [
                {
                    "type": "text",
                    "text": "참고할 문서:\n\n[긴 문서 내용]",
                    "cache_control": {"type": "ephemeral"}  # 캐시 포인트 2
                },
                {
                    "type": "text",
                    "text": "질문: 이 문서에서 인증 방법을 알려줘"
                    # 캐싱 안 함 — 매번 다른 질문
                }
            ]
        }
    ]
)
```

시스템 프롬프트와 참고 문서는 캐시, 실제 질문은 캐시 없이 전달.

## OpenAI에서 Prompt Caching

OpenAI는 자동으로 프롬프트 캐싱을 처리한다. 설정 없이 같은 내용이 반복되면 자동 캐싱.

조건:

- 프롬프트가 1024 토큰 이상
- 직전에 동일한 prefix가 포함된 요청이 있었을 것

```python
from openai import OpenAI

client = OpenAI()

# 긴 시스템 프롬프트가 반복되면 자동 캐싱
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {
            "role": "system",
            "content": "긴 시스템 프롬프트..." # 1024 토큰 이상이면 자동 캐싱
        },
        {"role": "user", "content": "질문"}
    ]
)

# 캐시 사용 확인
usage = response.usage
print(f"캐시 읽기: {usage.prompt_tokens_details.cached_tokens}")
```

OpenAI는 캐시를 명시적으로 지정하지 않아도 되는 대신, 캐시 여부를 완전히 제어하기 어렵다.

---

## 캐싱이 효과적인 패턴

**효과 큰 경우:**

- 긴 시스템 프롬프트가 매 요청에 포함될 때
- RAG에서 같은 문서 컨텍스트가 반복될 때
- 멀티턴 대화에서 이전 대화 내용이 쌓일 때
- 배치 처리에서 같은 기반 프롬프트를 반복할 때

**효과 작은 경우:**

- 프롬프트가 짧을 때 (1024 토큰 미만)
- 매 요청마다 내용이 크게 다를 때
- 캐시 TTL(보통 5분) 이내에 재요청이 없을 때

---

## 비용 계산 예시

월 10만 건의 API 요청, 요청당 시스템 프롬프트 2000 토큰:

```
캐싱 전:
2000 토큰 × 100,000 요청 = 2억 토큰
Claude Opus 기준 $3/1M 토큰 = $600/월

캐싱 후:
첫 요청: 2000 × 1.25 = 2500 (쓰기 비용)
나머지 99,999요청: 2000 × 0.1 = 200 토큰 × 99,999 ≈ 2000만 토큰
총 비용 ≈ $60/월 (90% 절감)
```

---

## 실습

API를 직접 쓰는 서비스가 있다면:

1. 매 요청에 반복되는 시스템 프롬프트를 찾는다
2. Anthropic이면 `cache_control` 추가, OpenAI면 프롬프트 길이 확인
3. 응답의 `cache_read_input_tokens`를 로깅해서 캐시 히트율 확인

RAG + Prompt Caching을 함께 쓰면 비용을 90% 이상 줄이는 것도 가능하다.

## 시리즈를 마치며

12편에 걸쳐 다룬 내용을 정리한다.

---

### 1편 — AI를 제대로 쓰는 핵심 3원칙

토큰 낭비, 맥락 분산, 반복 설명. 이 세 가지가 AI 사용 효율을 떨어뜨리는 주범이다. 긴 대화는 새 창으로, 같은 내용은 한 번만 설명하고, 맥락은 집중해서 유지하는 것. 기술적인 내용 이전에 습관의 문제다.

### 2편 — AI 세팅 고정

매번 같은 설명을 반복하지 않는 법. System Prompt, CLAUDE.md, Claude Projects를 활용하면 "나는 TypeScript를 쓰고, 한국어로 대답해줘"를 한 번만 설정하면 된다. 세팅 자체가 맥락이 된다.

### 3편 — MCP

프롬프트로 설명하는 것과 도구로 연결하는 것은 다르다. MCP를 쓰면 AI가 파일을 직접 읽고, 브라우저를 열고, DB를 조회할 수 있다. AI가 텍스트를 생성하는 존재에서 실제로 무언가를 하는 존재로 바뀐다.

### 4편 — MCP 실전

Claude Code와 Cursor에서 MCP 서버를 실제로 연결하는 방법. Filesystem, GitHub, 브라우저 자동화 같은 서버를 붙이고, 필요하면 커스텀 서버를 직접 만드는 것까지. 설정 파일 한 줄로 AI 능력이 확장된다.

### 5편 — Function Calling

AI가 텍스트 대신 함수 이름과 인자를 반환하게 만드는 구조. "날씨 알려줘"를 받으면 `get_weather(location="Seoul")`을 반환하고, 코드가 실제로 함수를 실행한다. AI와 코드 사이의 경계를 설계하는 방법이다.

### 6편 — Structured Output

JSON 스키마로 AI 응답 형태를 고정하면 파싱 없이 바로 쓸 수 있다. Function Calling이 "무엇을 실행할지"를 전달하는 구조라면, Structured Output은 "데이터를 어떤 형태로 받을지"를 고정하는 구조다. 둘을 언제 쓰는지 기준도 이 편에서 정리했다.

### 7편 — AI 에이전트

에이전트는 루프 + 도구 + 판단으로 이루어진다. 한 번 실행하고 끝나는 게 아니라, 결과를 보고 다음 행동을 결정하며 목표에 도달할 때까지 반복한다. Claude Code와 Cursor에서 에이전트 모드를 실제로 쓰는 방법도 다뤘다.

### 8편 — 서브에이전트 / 멀티에이전트

오케스트레이터가 서브에이전트에게 작업을 나눠주는 구조. 코드 리뷰를 예로 들면, 오케스트레이터가 "보안 검토", "성능 검토", "스타일 검토"를 각각 다른 에이전트에게 동시에 맡기고 결과를 취합한다. 병렬 처리와 역할 분리가 핵심이다.

### 9편 — 에이전트 직접 만들기

LangChain, OpenAI Agents SDK, Claude Agent SDK 세 가지 방법으로 코드 리뷰 에이전트를 직접 만들어봤다. 각각 추상화 수준과 유연성이 다르다. 어떤 도구를 선택할지는 프로젝트 규모와 제어 수준 필요에 따라 다르다.

### 10편 — 훅과 플러그인

Claude Code의 훅 시스템으로 AI 동작을 가로채고 제어하는 방법. 위험한 명령을 실행 전에 차단하거나, 특정 작업이 끝나면 자동으로 알림을 보내거나, 커스텀 워크플로우를 심을 수 있다. AI를 쓰는 방식 자체를 커스터마이징하는 레이어다.

### 11편 — RAG

컨텍스트 한계를 넘어 수천 페이지 문서를 AI가 참조하게 만드는 방법. 임베딩으로 문서를 벡터로 변환하고, 질문과 가장 관련 있는 부분만 뽑아서 AI에게 전달한다. 모든 걸 컨텍스트에 넣는 대신, 필요한 것만 꺼내 쓰는 구조다.

### 12편 — Prompt Caching

반복되는 시스템 프롬프트나 문서를 캐싱해서 비용을 90%까지 줄이는 방법. 같은 내용을 매 요청마다 다시 처리하는 게 아니라, 첫 번째 요청에서 캐시에 저장하고 이후 10% 비용으로 재사용한다.
