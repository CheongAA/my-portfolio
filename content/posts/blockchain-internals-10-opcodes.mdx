지난 편에서 Solidity 코드가 컴파일되면 바이트코드가 되고, 그 바이트코드는 EVM 옵코드의 시퀀스라고 했다.

옵코드는 EVM의 명령어 집합이다. 150개 정도가 있다. 오늘은 자주 쓰이는 것들과, 보안에서 중요한 CALL 계열을 집중적으로 살펴본다.

---

## 기본 스택 연산

### PUSH1 ~ PUSH32
스택에 값을 올린다. 숫자는 뒤에 오는 바이트 수를 의미한다.

```
PUSH1 0x05  → 스택: [5]
PUSH2 0x01F4 → 스택: [5, 500]
```

### POP
스택 최상위 값을 제거한다.

```
스택: [3, 5]
POP  → 스택: [3]
```

### DUP1 ~ DUP16
스택의 n번째 값을 복사해서 최상위에 올린다.

```
스택: [3, 5]
DUP1 → 스택: [5, 3, 5]  (최상위 5를 복사)
```

### SWAP1 ~ SWAP16
스택 최상위와 n번째를 교환한다.

## 산술 연산

```
ADD  : 스택 최상위 두 값을 꺼내 더한 뒤 결과 push
SUB  : 빼기
MUL  : 곱하기
DIV  : 나누기 (정수 나눗셈, 소수점 없음)
MOD  : 나머지
EXP  : 거듭제곱 (비쌈 — 지수에 비례해서 가스 증가)
```

이더리움은 uint256 연산이라 오버플로우가 그냥 wrapping된다. Solidity 0.8.0 이전에는 수동으로 SafeMath를 써야 했던 이유다. 0.8.0 이후엔 컴파일러가 자동으로 오버플로우 체크를 추가한다.

## 스토리지 연산: 가장 비싼 명령어들

### SLOAD
스토리지에서 값을 읽는다.

```
PUSH1 0x00  → 스토리지 슬롯 0을 스택에 push
SLOAD       → 슬롯 0의 값을 읽어서 스택에 push
```

비용: Cold access(처음 읽기) 2,100 가스 / Warm access(이미 읽은 슬롯) 100 가스.

### SSTORE
스토리지에 값을 쓴다.

```
PUSH1 0x01  → 저장할 값 1
PUSH1 0x00  → 스토리지 슬롯 0
SSTORE      → 슬롯 0에 1을 저장
```

비용: 새 값 쓰기 20,000 가스 / 기존 값 수정 2,900 가스 / 0으로 설정 시 환불.

SSTORE가 ADD(3 가스)보다 수천 배 비싼 이유 — 스토리지 쓰기는 영구적으로 모든 노드의 데이터베이스에 기록되기 때문이다. 이 비용은 "내가 모든 노드에게 평생 이 데이터를 저장해달라고 요청하는 대가"다.

## 메모리 연산

### MSTORE
메모리 오프셋에 32바이트 값을 쓴다. SSTORE보다 훨씬 저렴하다.

### MLOAD
메모리 오프셋에서 32바이트를 읽는다.

메모리는 실행 중에만 존재한다. 트랜잭션이 끝나면 사라진다.

## CALL 계열: 가장 중요하고 복잡한 명령어

컨트랙트가 다른 컨트랙트를 호출하는 세 가지 방법이 있다. 이 차이를 이해하는 게 스마트 컨트랙트 보안의 핵심이다.

### CALL
다른 컨트랙트를 **그 컨트랙트의 컨텍스트**에서 호출한다.

```
컨트랙트 A가 컨트랙트 B를 CALL로 호출
→ B의 코드가 실행됨
→ B의 storage에 변경이 생김
→ msg.sender는 A
→ msg.value로 ETH 전송 가능
```

일반적인 함수 호출. ERC-20 transfer를 호출하면 그 토큰 컨트랙트의 잔액 장부(storage)가 바뀐다.

### DELEGATECALL
다른 컨트랙트의 코드를 **내 컨텍스트**에서 실행한다.

```
컨트랙트 A가 컨트랙트 B를 DELEGATECALL로 호출
→ B의 코드가 실행되지만
→ A의 storage에 변경이 생김  ← 핵심 차이!
→ msg.sender는 원래 호출자
→ msg.value 변경 없음
```

마치 "B의 코드를 복사해서 A에서 실행하는 것"과 같다.

이게 **프록시 패턴**의 기반이다. 업그레이드 가능한 컨트랙트는 프록시(데이터 저장)와 로직 컨트랙트로 분리되고, 프록시가 DELEGATECALL로 로직을 실행한다. 로직 컨트랙트를 교체하면 업그레이드가 된다.

보안 위험: DELEGATECALL을 쓸 때 storage 슬롯 레이아웃이 두 컨트랙트 간에 일치해야 한다. 불일치하면 엉뚱한 storage가 덮어써진다. 실제로 이 버그로 수백억 원이 해킹된 사례가 있다.

### STATICCALL
읽기 전용 호출. 상태를 변경하는 옵코드가 실행되면 즉시 revert.

```
STATICCALL로 호출하면
→ SSTORE 불가
→ CALL(ETH 전송 있는) 불가
→ CREATE 불가
```

Solidity의 `view`와 `pure` 함수가 외부에서 호출될 때 내부적으로 STATICCALL을 쓴다. 상태 변경 없음을 EVM 레벨에서 강제한다.

## 제어 흐름

### JUMP / JUMPI
JUMP는 무조건 분기, JUMPI는 조건부 분기다. if문, 반복문이 이걸로 구현된다.

JUMPDEST가 표시된 위치로만 점프할 수 있다. 임의의 바이트코드 위치로 점프해서 악용하는 걸 방지하기 위해서다.

### RETURN / REVERT
RETURN: 정상 종료, 반환값 지정
REVERT: 실패 종료, 상태 변경 롤백, 에러 메시지 반환

## 실습: Remix에서 바이트코드와 옵코드 확인

1. remix.ethereum.org 접속
2. 간단한 컨트랙트 작성
3. 컴파일 후 좌측 "Compilation Details" 클릭
4. "BYTECODE" 항목에서 `object` 필드 = 배포 바이트코드
5. "OPCODES" 항목에서 역어셈블된 옵코드 시퀀스 확인

또는 `cast disassemble` (foundry 도구)으로 바이트코드를 옵코드로 변환할 수 있다.

```bash
cast disassemble 6080604052...
```
