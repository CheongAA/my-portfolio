지난 편에서는 코드를 어떻게 잘라서 보낼지, 원하는 형식을 어떻게 명시할지를 다뤘다.

이번 편은 다른 종류의 비효율을 다룬다. 내용이 아니라 **반복**이다.

---

## 매번 설명하고 있는 것들

새 대화를 시작할 때마다 이걸 쓰고 있지 않은지.

```
이 프로젝트는 Next.js 15 App Router야. TypeScript strict mode 쓰고,
Tailwind CSS v4로 스타일링해. 컴포넌트는 무조건 'use client' 붙이지 말고
서버 컴포넌트 우선으로 설계해줘. 함수형으로 써줘. 주석은 최소한으로.
```

이걸 매번 첫 메시지에 붙이거나, 아니면 안 붙이고 나중에 엉뚱한 답변을 받고 나서야 "아, 이건 Next.js야"라고 추가 설명한다.

둘 다 비효율이다.

---

## CLAUDE.md — 프로젝트 규칙을 파일로 고정하기

Claude Code를 쓴다면 이 방법이 제일 직접적이다.

프로젝트 루트에 `CLAUDE.md` 파일을 만든다. Claude는 대화를 시작할 때 이 파일을 자동으로 읽는다. 매 대화마다 설명하지 않아도 된다.

내가 실제로 쓰는 파일 구조는 이렇다.

```markdown
# 프로젝트 규칙

## 기술 스택

- Next.js 15 App Router
- TypeScript (strict mode)
- Tailwind CSS v4
- next-intl (i18n)

## 코딩 규칙

- 서버 컴포넌트 우선. 'use client'는 꼭 필요한 경우에만
- 함수형 컴포넌트만 사용
- 주석은 로직이 명확하지 않을 때만
- 타입은 interface 우선, type은 유니온/인터섹션에만

## 금지 패턴

- any 타입 사용 금지
- useEffect로 데이터 페칭 금지 (서버 컴포넌트에서 처리)
- 인라인 스타일 금지

## 경로 규칙

- 컴포넌트: src/components/
- 페이지: src/app/[locale]/
- i18n 키: src/i18n/locales/
```

이게 있으면 "Next.js야, TypeScript야" 설명을 다시 할 필요가 없다. Claude가 이미 알고 있다. 규칙에 어긋나는 코드를 제안하면 알아서 수정하거나 먼저 물어본다.

## 시스템 프롬프트 — Claude Projects, Custom GPT

Claude Code 외에 ChatGPT나 Claude 웹 인터페이스를 쓴다면 이 방법이다.

**Claude Projects**와 **Custom GPT** 둘 다 "시스템 프롬프트"를 미리 설정할 수 있다. 이 안에 프로젝트 규칙을 넣어두면 그 프로젝트 안의 모든 대화에 자동 적용된다.

### Claude Projects 설정 방법

1. claude.ai 접속 후 왼쪽 사이드바에서 **Projects → New Project**
2. 프로젝트 이름 입력 후 생성
3. 프로젝트 화면 오른쪽 상단 **Project instructions** 클릭
4. 텍스트 에디터에 규칙 입력 후 저장

이후로 이 프로젝트 안에서 여는 모든 대화에 자동 적용된다.

### Custom GPT 설정 방법

1. chatgpt.com 접속 후 왼쪽 사이드바에서 **New Project → Create**
2. 프로젝트 이름 입력 후 생성
3. 프로젝트 화면 오른쪽 상단 **More → Project instructions** 클릭
4. 텍스트 에디터에 규칙 입력 후 저장

북마크해두고 이 프로젝트 관련 질문은 여기서 하면 된다.

### 시스템 프롬프트 예시

"포트폴리오 프로젝트" 전용으로 이렇게 써두면 된다.

```
너는 이 포트폴리오 프로젝트의 개발 파트너야.

기술 스택: Next.js 15 App Router, TypeScript strict, Tailwind CSS v4, next-intl
배포 환경: GitHub Pages (static export, basePath: /my-portfolio)

코딩 규칙:
- 서버 컴포넌트 우선. 'use client'는 꼭 필요한 경우에만
- Framer Motion으로 애니메이션
- 컴포넌트는 src/components/ 하위에 위치
- 주석 최소화

답변할 때 이 맥락을 항상 유지하고, 이 스택에 맞는 코드를 줘.
모르는 게 있으면 추측하지 말고 물어봐.
```

새 대화를 열어도 설명을 다시 할 필요가 없다. 프로젝트 전환 시엔 해당 프로젝트/GPT로 바꿔서 열기만 하면 된다.

## 맥락 재사용 = 토큰 절약 + 일관성

이게 단순히 편의 문제만은 아니다.

매 대화마다 기술 스택 설명을 첫 메시지에 붙이면, 그 설명만큼 토큰이 소비된다. 하루에 대화를 10번 한다면 같은 설명을 10번 보내는 거다. 쌓이면 꽤 된다.

더 중요한 건 **일관성**이다. 시스템 프롬프트나 CLAUDE.md에 규칙을 고정해두면, 모든 대화에서 같은 기준으로 코드를 받는다. 어떤 날은 `interface`로 타입을 잡아주고, 어떤 날은 `type`으로 잡아주는 들쭉날쭉함이 사라진다.

맥락을 재사용하는 것은 비용 절약이면서 동시에 답변 품질을 고르게 유지하는 방법이다.

## 세팅 파일 관리 팁

한 가지 주의할 점이 있다.

CLAUDE.md나 시스템 프롬프트는 처음 만들고 잊어버리면 안 된다. 프로젝트가 바뀌면 규칙도 바뀐다. 새 라이브러리를 추가했거나, 규칙을 수정했다면 파일도 같이 업데이트해야 한다.

실용적인 관리 방법:

- **CLAUDE.md는 git에 함께 커밋** — 프로젝트 문서의 일부로 관리
- **규칙이 바뀔 때 파일도 같이 수정** — 코드 변경과 동시에
- **너무 길게 쓰지 말 것** — AI가 읽는 파일도 길어지면 핵심이 묻힌다. 핵심 규칙만 간결하게

## 실습

지금 자주 쓰는 프로젝트가 있다면, 이걸 만들어보자.

1. 프로젝트 루트에 `CLAUDE.md` 파일 생성
2. 기술 스택, 코딩 규칙, 금지 패턴을 각 섹션으로 정리
3. 다음 대화에서 첫 메시지에 설명 없이 바로 질문해보기

Claude Code가 아니라 웹 인터페이스를 쓴다면 Claude Projects나 Custom GPT에서 시스템 프롬프트를 설정하는 것으로 대신할 수 있다.

설명 없이 질문했는데 기술 스택에 맞는 코드가 바로 나오면 성공이다.
