스마트 컨트랙트를 배포하면 그 코드가 블록체인에 저장된다고 했다. 그리고 누군가 그 컨트랙트를 호출하면 코드가 "실행"된다.

그런데 수천 개의 노드가 각각 다른 하드웨어, 다른 운영체제를 쓰고 있는데, 어떻게 모두가 똑같은 결과를 얻을 수 있을까? 그 답이 **EVM(Ethereum Virtual Machine)**이다.

---

## 왜 가상 머신이 필요한가

컴퓨터 프로그램은 보통 특정 운영체제와 CPU 아키텍처에 종속된다. Windows용 프로그램이 macOS에서 안 돌아가고, x86 코드가 ARM CPU에서 안 돌아간다.

블록체인에서 이게 문제가 되는 이유는 **결정론적 실행**이 필수이기 때문이다.

결정론적 실행이란 — 어떤 노드에서 실행해도 같은 입력에 대해 반드시 같은 결과가 나와야 한다는 것이다.

왜? 모든 노드가 같은 계산을 해서 같은 stateRoot를 만들어내야 합의가 이루어지기 때문이다. 노드마다 결과가 다르면 어떤 stateRoot가 "맞는" 것인지 결정할 수 없다.

그래서 특정 하드웨어나 OS에 의존하지 않는, 명세가 완전히 정해진 가상 머신이 필요했다. 그게 EVM이다.

EVM 명세를 따르는 클라이언트라면 — 리눅스든 윈도우든, 인텔 CPU든 ARM이든 — 같은 코드를 같은 방식으로 실행해서 반드시 같은 결과를 낸다.

---

## 스택 기반 머신

EVM은 **스택(Stack) 기반** 가상 머신이다.

대부분의 현대 프로세서는 레지스터 기반이다. 레지스터라는 빠른 임시 저장소가 있고, 연산에 필요한 값을 레지스터에 올려두고 쓴다.

스택 기반은 다르다. LIFO(Last In, First Out) 스택을 중심으로 모든 연산이 이루어진다. 값을 스택에 쌓고(PUSH), 연산이 필요하면 스택 위에서 꺼내서(POP) 계산한다.

예를 들어 `3 + 5`를 계산한다면:

```
PUSH 3   → 스택: [3]
PUSH 5   → 스택: [3, 5]
ADD      → 스택에서 5와 3을 꺼내 더한 뒤 결과를 다시 push
           스택: [8]
```

레지스터 기반보다 느리지만, 명세가 단순하고 다양한 환경에서 동일하게 구현하기 쉽다. 블록체인의 결정론적 실행 요구사항에 잘 맞는다.

EVM 스택은 최대 1024개의 항목을 담을 수 있다. 각 항목은 256비트(32바이트)다.

## EVM의 네 가지 데이터 공간

EVM에서 코드가 실행될 때 데이터를 저장할 수 있는 공간이 네 가지 있다.

### 1. Stack
앞서 설명한 스택. 임시 계산에 쓰인다. 비용이 거의 없다.

### 2. Memory
실행 중에만 존재하는 임시 저장소. 배열처럼 인덱스로 접근한다. 크기를 늘릴수록 비용이 2차함수적으로 늘어난다.

```solidity
// Solidity에서 memory 키워드
function foo(uint256[] memory arr) public { ... }
```

### 3. Storage
영구적인 저장소. 컨트랙트 계정의 storageRoot에 담긴다. 블록체인에 영구히 기록되므로 비용이 가장 비싸다. (SSTORE 명령어: 20,000 가스)

```solidity
// Solidity에서 storage는 상태 변수
uint256 public count;  // 이게 Storage에 저장됨
```

### 4. Calldata
함수를 호출할 때 전달하는 읽기 전용 데이터. Memory보다 저렴하다. 함수 인자가 여기 들어온다.

## Solidity → 바이트코드 → 옵코드

개발자가 Solidity로 코드를 작성하면 컴파일 과정을 거쳐 EVM이 실행하는 **바이트코드**가 된다.

```
Solidity 소스코드
       ↓ solc (컴파일러)
   바이트코드 (Bytecode)    ← 블록체인에 저장되는 것
       ↓ 해석
옵코드 (Opcode) 시퀀스       ← EVM이 실제로 실행하는 것
```

바이트코드는 이런 식으로 생겼다.

```
6080604052348015600f57600080fd5b5060...
```

16진수 문자열이다. 알아보기 어렵다. 이걸 옵코드로 변환하면:

```
PUSH1 0x80
PUSH1 0x40
MSTORE
CALLVALUE
DUP1
ISZERO
PUSH1 0x0f
JUMPI
...
```

옵코드는 EVM의 명령어 집합이다. 다음 편에서 자세히 다룬다.

## 간단한 예: Solidity 함수가 바이트코드가 되는 과정

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract Counter {
    uint256 public count;

    function increment() public {
        count += 1;
    }
}
```

`increment()` 함수를 EVM 관점에서 보면 이렇게 동작한다.

```
1. SLOAD   → Storage에서 count 값 읽어서 스택에 push
2. PUSH1 1 → 숫자 1을 스택에 push
3. ADD     → 스택에서 두 값 꺼내 더함
4. SSTORE  → 결과를 Storage에 저장
```

단순해 보이지만 실제 컴파일된 바이트코드에는 함수 셀렉터 분기, 오버플로우 체크, 이벤트 발행 등이 추가되어 훨씬 길다.

## EVM의 한계: 외부 세계에 접근 불가

EVM은 샌드박스 환경이다. 다음이 불가능하다.

- 인터넷에서 데이터 가져오기
- 시스템 클락 읽기 (블록 timestamp만 사용 가능)
- 난수 생성 (진짜 무작위는 불가능)
- 다른 블록체인과 직접 통신

이런 외부 데이터가 필요할 때 쓰는 게 오라클(Oracle)이다. 외부 데이터를 체인 위로 가져오는 브릿지 역할을 한다. (DeFi 시리즈에서 자세히)
