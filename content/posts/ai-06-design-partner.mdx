지난 편에서는 대화를 언제 이어가고 언제 끊을지, 그리고 수정 요청을 어떻게 해야 효과가 있는지를 다뤘다.

이번 편은 그보다 더 앞 단계로 간다. 구현 시작 전이다.

---

## "만들어줘" 전에 무엇이 빠져 있는가

보통 이렇게 한다.

```
로그인 기능 만들어줘. JWT 써서.
```

AI는 코드를 준다. 나는 받아서 붙여넣는다. 실행해보니 동작한다. 그런데 며칠 뒤, 요구사항이 바뀐다. 소셜 로그인을 추가해야 한다. 토큰 저장 위치를 바꿔야 한다. 기존 구조가 맞지 않아서 처음부터 다시 짜야 한다.

문제는 코드가 아니었다. **설계였다.**

AI에게 바로 구현을 요청하면, AI는 가장 일반적인 방식으로 만든다. 내 프로젝트의 구체적인 제약, 앞으로의 확장 가능성, 다른 부분과의 연결을 AI는 모른다. 내가 알려주지 않았으니까.

---

## 설계 단계에서 AI를 쓰는 방식

구현 전에 먼저 이렇게 물어본다.

```
로그인 기능을 만들려고 해. JWT 방식으로.
코드 짜기 전에 설계 먼저 같이 생각해보고 싶어.

현재 상황:
- Next.js 15 App Router
- 나중에 소셜 로그인(Google, GitHub) 추가 가능성 있음
- 모바일도 지원해야 함

어떤 방식들이 있는지, 각각의 트레이드오프를 정리해줘.
```

이러면 AI가 구현 코드 대신 **선택지와 트레이드오프**를 준다.

예를 들면 이런 식이다:

```
방식 1: httpOnly 쿠키에 JWT 저장
장점: XSS 공격 방어. 브라우저 JS에서 접근 불가.
단점: CSRF 방어 추가 필요. 서버에서 쿠키 설정해야 함.

방식 2: localStorage에 저장
장점: 구현 단순. 클라이언트에서 바로 접근 가능.
단점: XSS 취약. 토큰이 노출될 수 있음.

방식 3: 메모리(zustand/context)에 저장
장점: 보안 최고. 탭 닫으면 자동 로그아웃.
단점: 페이지 새로고침 시 로그아웃됨. UX 나쁨.
```

이제 내가 선택한다. 소셜 로그인 추가 가능성이 있고 보안이 중요하니 방식 1로 간다고 결정한다. **이 선택은 내가 한다.** AI가 추천하는 방식을 그대로 따르는 게 아니라.

결정하고 나서 구현을 요청하면, 구조가 애초부터 맞게 나온다.

---

## "이 방식의 단점이 뭐야?"

설계 논의에서 가장 유용한 질문 중 하나다.

AI는 기본적으로 제안한 방식을 긍정적으로 설명하는 경향이 있다. "이 방식이 좋습니다"로 끝나는 경우가 많다. 직접 단점을 캐물어야 균형 잡힌 정보를 얻는다.

```
httpOnly 쿠키 방식으로 가기로 했는데,
이 방식의 단점이나 놓치기 쉬운 부분이 뭐야?
```

이러면 다르게 나온다.

```
주의할 점:
1. CSRF 토큰 처리 — 쿠키 기반 인증은 CSRF 공격에 노출.
   SameSite=Strict나 별도 CSRF 토큰 필요.

2. 서버리스 환경 주의 — Vercel Edge Functions에서
   Set-Cookie 헤더 처리가 제한될 수 있음.

3. 토큰 갱신 로직 — 액세스 토큰 만료 시 자동 갱신 처리를
   명시적으로 구현해야 함. 안 하면 사용자가 갑자기 로그아웃됨.
```

이게 나중에 터질 수 있는 지뢰들이다. 미리 알고 설계에 반영할 수 있다.

## 실전: 설계 대화 흐름 예시

실제로 어떻게 진행되는지 한 흐름을 보여주면 이렇다.

**1단계: 요구사항 정리**

```
북마크 기능을 추가하려고 해.

조건:
- 로그인 없이도 저장 가능 (로컬 저장)
- 로그인 후엔 서버와 동기화
- 새 탭에서도 유지

어떤 구조가 적합할까?
```

**2단계: 선택지 받기**

AI가 localStorage + 서버 DB 조합, IndexedDB 방식, 쿠키 방식 등을 정리해준다. 각각의 장단점과 함께.

**3단계: 단점 캐내기**

```
localStorage + 서버 DB 조합으로 가려는데, 이 방식에서 동기화할 때
충돌 처리는 어떻게 해야 해? 로컬 데이터랑 서버 데이터가 다를 때.
```

**4단계: 결정 후 구현 요청**

```
last-write-wins 방식으로 처리하기로 했어.
이 결정을 바탕으로 북마크 기능 구현해줘.

조건:
- 비로그인: localStorage만 사용
- 로그인 후: 서버에 업로드, 이후 서버 기준으로 동기화
- 충돌 시: 서버 데이터 우선
```

이 흐름에서 포인트는 하나다. **내가 결정하는 것이 설계다.** AI는 선택지와 정보를 준다. 결정은 내가 한다.

---

## 왜 이게 더 효율적인가

처음에는 설계 단계가 시간 낭비처럼 느껴질 수 있다. 바로 코드 받아서 쓰는 게 빠를 것 같다.

실제로는 반대다.

구현하고 나서 설계가 틀렸다는 걸 알면, 전체를 다시 짜야 한다. 그게 더 많은 시간을 잡아먹는다. 설계 단계에서 30분 쓰는 게, 구현 후 재작업 2시간보다 낫다.

Ethan Mollick이 말한 "항상 켜둔 동료"를 여기서 활용하는 방식이 이거다. 구현 파트너가 아니라 **설계 파트너로 먼저 쓰는 것.** AI 초안(선택지)을 받고, 내가 판단하고, 그 다음 구현으로 넘어간다.

---

## 설계 대화를 이어가기 좋은 질문들

익혀두면 자주 쓰게 되는 것들이다.

| 상황 | 질문 |
|------|------|
| 선택지를 모를 때 | "이 기능을 구현하는 방법 3가지와 각각의 트레이드오프를 알려줘" |
| 결정 후 리스크 확인 | "이 방식을 선택했을 때 나중에 문제가 될 수 있는 게 뭐야?" |
| 구조 검토 | "이 설계에서 내가 놓친 게 있어?" |
| 확장성 확인 | "나중에 [기능]을 추가하려면 지금 설계에서 뭘 고려해야 해?" |
| 비교 정리 | "A방식과 B방식, 내 상황에서 어떤 게 나아?" |

## 실습

지금 구현하려는 기능이 하나 있다면, 바로 코드를 요청하기 전에 이렇게 해보자.

1. 기능의 요구사항을 3줄로 정리한다 (지금 상황, 필요한 것, 나중 확장 가능성)
2. "코드 짜기 전에 설계 먼저 얘기해보고 싶어 — 어떤 방식들이 있어?" 라고 묻는다
3. 선택지를 받은 뒤 "이 방식의 단점이 뭐야?"를 한 번 더 물어본다
4. 내가 직접 방식을 고른다
5. 그 다음에 구현을 요청한다
