지난 편에서 이더리움은 EVM 위에서 코드를 실행하는 플랫폼이라고 했다. 그 코드가 바로 스마트 컨트랙트다. 블록체인에서 가장 핵심이 되는 개념이다.

---

## 자판기 비유

스마트 컨트랙트를 이해하는 가장 쉬운 비유는 자판기다.

자판기에 500원을 넣고 콜라 버튼을 누르면 콜라가 나온다. 직원이 없어도 된다. 점주에게 전화해서 "콜라 주세요"라고 할 필요도 없다. **조건(500원 + 버튼)이 충족되면 결과(콜라)가 자동으로 실행된다.** 아무도 중간에 끼어들 수 없다.

스마트 컨트랙트도 같다. "A가 B에게 1 ETH를 전송하면, 자동으로 집 소유권이 이전된다" 같은 조건을 코드로 쓴다. 조건이 충족되면 이더리움 네트워크가 자동으로 실행한다. 변호사도, 공증인도, 은행도 필요 없다.

---

## 일반 계약서 vs 스마트 컨트랙트

부동산 거래를 생각해보자.

일반 계약:
1. 계약서 작성 (오류 가능)
2. 변호사 검토 (비용 발생)
3. 공증 (시간 소요)
4. 은행 에스크로 (수수료 발생)
5. 소유권 이전 등기 (관공서 방문)
6. 분쟁 발생 시 → 법원

스마트 컨트랙트:
1. 코드로 조건 작성 (누구나 볼 수 있음)
2. 블록체인에 배포
3. 조건 충족 → 즉시 자동 실행
4. 분쟁 여지 없음 (코드가 실행됐으면 그게 끝)

"코드가 법이다(Code is Law)"라는 말이 여기서 나왔다. 법원이 아니라 코드가 계약을 집행한다.

---

## 어떻게 동작하나

스마트 컨트랙트는 트랜잭션에 의해 실행된다.

```
EOA(지갑) → 트랜잭션 전송 → 컨트랙트 주소
                              ↓
                         EVM이 코드 실행
                              ↓
                    상태 변경 (블록체인에 기록)
                              ↓
                      결과 반환 / 이벤트 발생
```

컨트랙트는 스스로 실행되지 않는다. 반드시 외부 트랜잭션이 있어야 시작된다. 트랜잭션은 EOA(일반 지갑)나 다른 컨트랙트에서 보낼 수 있다.

컨트랙트가 다른 컨트랙트를 호출하는 것도 가능하다. 이걸 통해 복잡한 DeFi 시스템이 구성된다. Uniswap에서 토큰을 스왑하면 실제로는 여러 컨트랙트가 연쇄적으로 실행된다.

---

## 한 번 배포하면 수정 불가능하다

스마트 컨트랙트는 블록체인에 배포되면 **영구적으로 변경 불가능**하다.

왜? 블록체인의 불변성 때문이다. 컨트랙트 코드는 EVM 바이트코드로 변환돼 블록에 저장된다. 전 세계 수만 개의 노드가 복사본을 갖는다. 어느 한 노드가 코드를 수정해도 나머지 노드들과 맞지 않아 무효처리된다.

배포한 개발자도, 운영사도, 어떤 정부도 배포된 코드를 바꿀 수 없다.

**이게 장점이자 단점이다.**

장점:
- 서비스 운영자가 규칙을 몰래 바꿀 수 없다
- 사용자는 코드를 보고 정확히 어떻게 동작하는지 알 수 있다
- 누구도 실행을 막을 수 없다 ("검열 저항성")

단점:
- 버그가 있어도 수정이 안 된다
- 초기 설계가 매우 중요하다
- 보안 감사 없이 배포하면 위험하다

## The DAO 해킹: 불변성의 역설

2016년, 이더리움 역사에서 가장 큰 사건이 있었다.

"The DAO"라는 분산 투자 펀드 컨트랙트에 약 1억 5천만 달러 규모의 ETH가 모였다. 그런데 스마트 컨트랙트에 **재진입 공격(Reentrancy Attack)** 취약점이 있었다. 해커는 이를 이용해 약 6천만 달러 상당의 ETH를 탈취했다.

코드가 법이라면, 해킹도 법적으로는 "코드가 허용한 동작"이다. 하지만 커뮤니티는 이를 받아들일 수 없었다.

해결책은 하나였다. **이더리움 자체를 하드포크하는 것.** 블록체인의 역사를 특정 시점으로 되돌려 해킹 트랜잭션을 없애는 것이다.

대부분의 커뮤니티가 하드포크에 동의했고, 오늘날의 이더리움(ETH)이 됐다. 하드포크를 거부한 소수는 원본 체인을 유지했고, 이게 이더리움 클래식(ETC)이다.

이 사건은 "코드가 법이다"라는 명제에 의문을 제기했다. 코드의 결함으로 인한 결과까지 법으로 봐야 하는가? 이더리움 커뮤니티의 답은 "아니다"였다.

---

## 업그레이드 가능한 컨트랙트 (프록시 패턴)

실전에서는 버그 수정과 기능 추가가 필요하다. 불변성 원칙을 지키면서도 이를 가능하게 하는 패턴이 **프록시 패턴**이다.

```
사용자 → 프록시 컨트랙트 (주소 고정)
              ↓ delegatecall
         로직 컨트랙트 (교체 가능)
```

프록시 컨트랙트는 고정된 주소를 가지고, 실제 로직은 별도 컨트랙트에 있다. 버그 수정이 필요하면 로직 컨트랙트를 새로 배포하고, 프록시가 가리키는 주소를 바꾼다.

다만 이 방식은 "프록시 오너가 마음대로 로직을 바꿀 수 있다"는 중앙화 리스크가 생긴다. 트레이드오프다.

OpenZeppelin이 검증된 프록시 패턴 구현체를 제공한다. 입문 단계에서는 일단 이 패턴이 존재한다는 정도만 알면 된다.

---

## 실제 쓰이는 사례

**토큰 발행 (ERC-20)**

"이 컨트랙트를 통해서만 토큰을 발행하고 전송할 수 있다." 컨트랙트가 각 주소의 잔액을 추적하는 장부 역할을 한다. USDC, WETH, UNI 같은 모든 이더리움 기반 토큰이 ERC-20 컨트랙트다.

**NFT (ERC-721)**

"이 ID를 가진 NFT는 이 주소가 소유한다." 소유권을 컨트랙트가 기록한다. OpenSea에서 NFT를 사고팔 때 실제로는 컨트랙트의 소유권 정보가 업데이트되는 것이다.

**탈중앙화 거래소 (Uniswap)**

"유동성 풀에 두 토큰을 넣으면 수학 공식(x * y = k)으로 가격을 계산해 교환해준다." 중간에 오더북도, 매칭 엔진도, 직원도 없다. 순수하게 코드가 실행된다. Uniswap v3은 하루 수십억 달러의 거래를 처리한다.

**대출 프로토콜 (Aave, Compound)**

"담보를 넣으면 다른 자산을 빌릴 수 있다. 담보 가치가 일정 비율 아래로 떨어지면 자동으로 청산된다." 은행 없는 대출 시스템이다.

**투표 시스템 (DAO)**

"이 토큰을 가진 주소는 제안을 올리고 투표할 수 있다. 과반수가 동의하면 자동으로 실행된다." 결과를 조작할 수 없다.

## Solidity 코드로 보기

스마트 컨트랙트는 주로 **Solidity** 언어로 작성한다. JavaScript와 문법이 비슷해서 프론트엔드 개발자도 비교적 쉽게 읽힌다.

아래는 간단한 금고(Vault) 컨트랙트다.

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract SimpleVault {
    address public owner;
    mapping(address => uint256) public deposits; // 각 주소별 예치금

    event Deposited(address indexed user, uint256 amount);
    event Withdrawn(address indexed user, uint256 amount);

    constructor() {
        owner = msg.sender; // 배포한 사람이 오너
    }

    // ETH를 예치하는 함수 (payable = ETH를 받을 수 있음)
    function deposit() public payable {
        require(msg.value > 0, "Must send ETH");
        deposits[msg.sender] += msg.value;
        emit Deposited(msg.sender, msg.value);
    }

    // 예치한 ETH를 출금하는 함수
    function withdraw(uint256 amount) public {
        require(deposits[msg.sender] >= amount, "Insufficient balance");

        deposits[msg.sender] -= amount; // 먼저 상태 변경 (재진입 공격 방지)
        payable(msg.sender).transfer(amount);

        emit Withdrawn(msg.sender, amount);
    }

    // 전체 컨트랙트 잔액 조회
    function getBalance() public view returns (uint256) {
        return address(this).balance;
    }
}
```

코드를 읽으면 동작이 보인다.
- `deposit()`을 호출하면서 ETH를 보내면 `deposits[msg.sender]`가 증가한다
- `withdraw(amount)`를 호출하면 잔액 확인 후 ETH를 돌려준다
- 상태 변경(`deposits[msg.sender] -= amount`)을 ETH 전송보다 먼저 하는 건 재진입 공격 방지다

다음 편에서는 Solidity 문법을 프론트엔드 개발자 눈높이에 맞춰 체계적으로 다룬다.
