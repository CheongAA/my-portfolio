지난 편에서 블록 헤더에 `stateRoot`라는 필드가 있다고 했다. 이더리움에 존재하는 모든 계정의 현재 상태를 담은 트리의 루트 해시라고.

수백만 개의 계정 잔액이 해시 하나에 요약된다는 게 어떻게 가능한 걸까? 그리고 트랜잭션이 실행될 때마다 이 트리가 어떻게 업데이트될까?

---

## 먼저: 이더리움은 왜 UTXO를 안 쓰나

비트코인은 **UTXO(Unspent Transaction Output)** 모델을 쓴다. 잔액이라는 개념 자체가 없고, 아직 쓰지 않은 코인 조각들의 합이 곧 내 자산이다. 마치 지갑 안에 100원짜리, 500원짜리, 1000원짜리 동전이 있는 것처럼.

이더리움은 **계정(Account) 모델**을 쓴다. 각 주소마다 잔액이라는 숫자가 직접 저장된다. 은행 계좌처럼.

왜 이더리움은 UTXO를 버렸을까? 스마트 컨트랙트 때문이다.

컨트랙트는 상태를 저장해야 한다. 예를 들어 ERC-20 토큰 컨트랙트는 누가 얼마를 갖고 있는지 내부 장부를 유지한다. UTXO 모델에서는 이런 '지속되는 상태'를 표현하기가 굉장히 복잡해진다. 계정 모델은 상태를 자연스럽게 담을 수 있다.

---

## 이더리움의 두 계정 타입

이더리움의 계정은 두 종류다.

### EOA (Externally Owned Account)
우리가 MetaMask에서 쓰는 일반 지갑 주소다. 개인키로 제어한다. 코드가 없다.

상태 필드:
```
{
  nonce:   42,           // 이 주소가 보낸 트랜잭션 수
  balance: 1500000000000000000,  // 잔액 (wei 단위, 1.5 ETH)
  storageRoot: 0x00...  // 비어있음 (코드 없음)
  codeHash: 0x00...     // 비어있음 (코드 없음)
}
```

nonce는 트랜잭션 재생 공격을 막는다. 같은 서명된 트랜잭션을 두 번 제출해도, nonce가 이미 쓰인 번호라 두 번째 트랜잭션은 무시된다.

### CA (Contract Account)
스마트 컨트랙트 주소다. 개인키가 없다. 코드로 제어된다.

상태 필드:
```
{
  nonce:       1,        // 이 컨트랙트가 생성한 컨트랙트 수
  balance:     0,        // ETH 잔액 (컨트랙트도 ETH를 보유할 수 있음)
  storageRoot: 0x9f3a..., // 컨트랙트 내부 데이터의 Merkle Root
  codeHash:    0x7c81..., // 배포된 바이트코드의 해시
}
```

`storageRoot`가 흥미롭다. ERC-20 토큰 컨트랙트라면 여기 각 주소의 토큰 잔액이 담겨있다. 이 storageRoot 아래에 또 하나의 Patricia Merkle Trie(Storage Trie)가 있다.

---

## Patricia Merkle Trie란

일반 Merkle Tree는 데이터를 순서대로 쌓는다. 그런데 이더리움 계정은 순서가 없다. 주소를 키(key)로 계정 상태를 조회해야 한다.

이때 필요한 게 **Trie(트라이)** 자료구조다. 키를 경로로 삼아서 값을 찾는 트리다.

예를 들어 주소 `0x1a2b3c...`로 계정을 찾으려면, Trie에서 `1 → a → 2 → b → 3 → c → ...` 경로를 따라가면 된다.

여기에 **Patricia(경로 압축)**와 **Merkle(해시 체계)**를 합친 게 Patricia Merkle Trie다.

- **Patricia**: 공통 접두사가 있는 경로를 하나의 노드로 압축한다. 키가 `1a2b3c`와 `1a2b3d`라면, `1a2b3`까지는 한 노드로 묶고 마지막 `c/d`에서만 분기한다.
- **Merkle**: 각 노드가 자식 노드들의 해시를 포함한다. 루트 해시 하나로 전체 트리의 무결성을 보장한다.

```
                 [Root Node]
                  (stateRoot)
                /             \
        [Branch]               [Branch]
        /      \               /      \
  [0x1a...]  [0x2f...]   [0x8c...]  ...
```

---

## 왜 일반 Merkle Tree가 아니라 Patricia Merkle Trie인가

Merkle Tree는 데이터를 추가하거나 수정하면 루트까지 경로 전체를 재계산해야 한다. 트리 전체를 처음부터 다시 빌드하는 것과 다름없다.

이더리움에서 블록 하나가 생성될 때 수천 개의 트랜잭션이 처리되고, 그만큼 계정 상태가 변한다. 매 블록마다 수백만 계정 전체를 재계산하면 너무 느리다.

Patricia Merkle Trie는 변경된 계정의 경로만 업데이트한다. 나머지 경로는 그대로 재사용한다. 이걸 **구조적 공유(structural sharing)**라고 부른다.

```
변경 전:           변경 후 (계정 X만 변경):

[Root A]           [Root B]  ← 새 루트
 /    \              /    \
[L1]  [L2]        [L1']  [L2]  ← L2는 재사용
  \                  \
  [X]              [X']  ← X만 새로 생성
```

블록 하나가 처리될 때 이전 상태 트리와 새 상태 트리는 대부분의 노드를 공유한다. 변경된 경로만 새로 만든다.

---

## 트랜잭션 하나가 실행되면 stateRoot가 어떻게 변하나

Alice가 Bob에게 1 ETH를 보내는 트랜잭션이 실행되는 과정:

```
1. Alice 계정 조회
   → balance: 5 ETH, nonce: 42

2. 유효성 검사
   → 잔액 충분? (5 ETH > 1 ETH + 가스비) ✓
   → nonce가 올바른가? (트랜잭션 nonce = 42) ✓

3. 상태 변경
   → Alice: balance 5 ETH → 3.999... ETH, nonce 42 → 43
   → Bob:   balance 2 ETH → 3 ETH

4. 변경된 경로만 State Trie 업데이트
   → 새 stateRoot 계산
```

이 과정이 블록 안의 모든 트랜잭션에 대해 순서대로 실행된다. 마지막 트랜잭션 처리 후의 stateRoot가 블록 헤더에 기록된다.

## 실습: 특정 계정 상태를 직접 조회하기

`eth_getProof` RPC 메서드를 쓰면 특정 주소의 계정 상태와 State Trie Proof를 동시에 받을 수 있다.

```bash
curl https://mainnet.infura.io/v3/YOUR_KEY \
  -X POST \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "eth_getProof",
    "params": [
      "0xYOUR_ADDRESS",
      [],
      "latest"
    ],
    "id": 1
  }'
```

응답에 `balance`, `nonce`, `codeHash`, `storageHash`(storageRoot)가 나오고, `accountProof` 배열에 State Trie의 Merkle Proof 경로가 담겨있다.

## 네 개의 트리

정리하면, 이더리움에는 블록당 네 개의 Patricia Merkle Trie가 있다.

| Trie | Root 위치 | 내용 |
|------|---------|-----|
| State Trie | 블록 헤더 `stateRoot` | 모든 계정 상태 |
| Transaction Trie | 블록 헤더 `transactionsRoot` | 이 블록의 트랜잭션 목록 |
| Receipt Trie | 블록 헤더 `receiptsRoot` | 트랜잭션 실행 결과 |
| Storage Trie | CA 계정의 `storageRoot` | 컨트랙트 내부 데이터 |
