지난 편에서는 업계 리더들의 AI 활용 전략을 살펴봤다. 공통점은 하나였다 — 맥락은 내가 설계하고, 판단도 내가 한다.

이번 편부터는 기술로 들어간다. 어떻게 적게 보내면서 원하는 답을 받을 수 있는지.

---

## 코드를 줄이면 답이 나빠지지 않는가

처음엔 이렇게 생각했다. "더 많이 보낼수록 AI가 더 잘 알겠지."

틀렸다.

AI는 내가 보낸 텍스트 전체를 처리하지만, 주의가 고르게 분배되지 않는다. 입력이 길어질수록 중요한 부분이 다른 내용에 묻힌다. 500줄짜리 파일에서 실제 문제가 되는 코드는 10줄인데, 나머지 490줄이 자리를 차지하고 있다.

**관련 없는 코드는 노이즈다.** 노이즈를 줄이면 신호가 선명해진다.

---

## 뭘 잘라낼 수 있는가

코드를 AI에게 보내기 전에 이것부터 지운다.

**지워도 되는 것:**
- 에러와 관련 없는 다른 함수들
- 무관한 import 목록
- 주석 (AI에게 설명이 필요한 게 아니라면)
- 이미 동작하는 부분

**남겨야 하는 것:**
- 에러가 나는 함수 그 자체
- 그 함수가 쓰는 타입, 인터페이스
- 호출하는 쪽 코드 (어떻게 쓰이는지)
- 에러 메시지 전체

예를 들어 이런 파일이 있다고 하자.

```typescript
// utils.ts — 200줄

import { useState, useEffect, useCallback, useRef } from 'react'
import { formatDate, parseDate, diffDays } from './date-utils'
import { ApiClient } from './api'
import { UserProfile, Settings, Notification } from './types'

// 사용자 관련 함수들 (50줄)
export function getUserProfile() { ... }
export function updateSettings() { ... }
export function getNotifications() { ... }

// 날짜 관련 함수들 (80줄)
export function formatRelativeDate() { ... }
export function getDateRange() { ... }

// 여기서 에러가 남 (20줄)
export async function fetchUserData(userId: string): Promise<UserProfile> {
  const res = await ApiClient.get(`/users/${userId}`)
  return res.data  // 타입 에러 발생
}
```

AI에게 보낼 것은 이것만이다.

```typescript
// types.ts
interface UserProfile {
  id: string
  name: string
  email: string
}

interface ApiResponse<T> {
  data: T
  status: number
}

// 문제가 되는 함수
async function fetchUserData(userId: string): Promise<UserProfile> {
  const res = await ApiClient.get(`/users/${userId}`)
  return res.data  // Type 'ApiResponse<UserProfile>' is not assignable to type 'UserProfile'
}
```

200줄 → 20줄. 에러 메시지도 함께.

## 원하는 출력 형식을 명시하기

두 번째 기술이다. AI는 형식을 지정하지 않으면 알아서 길게 쓴다. 설명, 예시, 주의사항, 대안까지 붙인다. 그 중에 내가 필요한 건 핵심 코드 5줄인데.

원하는 게 뭔지 미리 말하면 그대로 나온다.

| 상황 | 추가할 말 |
|------|-----------|
| 코드만 필요할 때 | "코드만 줘, 설명 없이" |
| 짧은 설명이 필요할 때 | "2-3줄로 요약해줘" |
| 특정 형식을 맞춰야 할 때 | "주석 없이", "TypeScript로", "함수형으로" |
| 여러 방법이 궁금할 때 | "3가지 방법 각각 한 줄씩" |

before / after로 보면 차이가 명확하다.

**before (형식 미지정):**
```
fetchUserData 함수에서 타입 에러가 납니다. 고쳐주세요.

[코드]
```

받은 답변: 타입 에러의 원인 설명 3단락 + 수정 방법 2가지 + 각각의 장단점 + 권장 방법 + 추가 고려사항. 스크롤이 한참이다.

**after (형식 지정):**
```
fetchUserData 함수 타입 에러를 수정해줘. 수정된 함수 코드만 줘.

[코드]
[에러 메시지]
```

받은 답변:
```typescript
async function fetchUserData(userId: string): Promise<UserProfile> {
  const res = await ApiClient.get<ApiResponse<UserProfile>>(`/users/${userId}`)
  return res.data
}
```

끝. 바로 붙여넣고 확인했다.

## few-shot: 예시를 보여주면 형식이 따라온다

설명보다 빠른 방법이 있다. 예시를 직접 보여주는 것이다.

내가 원하는 출력이 어떤 형태인지 샘플을 하나 보여주면, AI는 그 형식을 그대로 따른다. 이걸 few-shot이라고 부른다.

예를 들어 여러 함수를 한꺼번에 정리하고 싶을 때.

```
아래 함수들을 이 형식으로 정리해줘:

예시:
함수명: getUserProfile
역할: userId로 사용자 정보를 가져옴
반환: UserProfile

정리할 함수들:
[코드]
```

"이 형식으로" 라고 말하는 것보다, 예시 하나 보여주는 게 훨씬 정확하다. 특히 테이블, 목록, 특정 JSON 구조 같은 걸 원할 때 유용하다.

---

## 실습: 코드 보내기 전 루틴

지금 작업 중인 코드에서 AI에게 물어볼 게 있다면, 보내기 전에 이 순서로 한 번 걸러본다.

1. **필요한 것만 추출** — 에러나는 함수 + 관련 타입만 남기고 나머지 삭제
2. **형식 명시** — 답변에서 뭘 원하는지 한 줄 추가 ("코드만", "3줄 요약" 등)
3. **에러 메시지 포함** — 에러가 있다면 메시지 전체를 복사해서 함께

처음엔 이 단계가 귀찮게 느껴질 수 있다. 근데 전처리에 30초 쓰면 답변에서 원하는 걸 찾느라 쓰는 3분을 아낀다.
