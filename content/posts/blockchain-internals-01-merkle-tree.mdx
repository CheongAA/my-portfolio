입문 시리즈에서 블록체인은 해시로 블록들을 연결한다고 했다.

그런데 이런 의문이 생겼다. 블록 하나에 트랜잭션이 수천 개씩 들어있는데, 그 수천 개를 어떻게 해시 하나로 요약하는 걸까? 그리고 "이 트랜잭션이 이 블록에 들어있다"는 걸 어떻게 빠르게 증명하는 걸까?

그 답이 바로 **Merkle Tree**다.

---

## 해시 함수 복습

Merkle Tree를 이해하려면 해시 함수부터 다시 짚고 가야 한다.

해시 함수는 어떤 데이터를 넣어도 항상 같은 길이의 출력값이 나오는 함수다. 이더리움은 keccak256을 쓴다.

```
keccak256("Alice → Bob 1 ETH")  → a3f7c2...
keccak256("Alice → Bob 2 ETH")  → 9d81e0...  ← 숫자 하나 달라졌더니 완전히 다른 값
```

두 가지 특성이 있다:

1. **같은 입력 → 항상 같은 출력**: 결정론적이라 검증에 쓸 수 있다.
2. **출력으로 입력을 역산할 수 없다**: 단방향 함수라 위변조 탐지에 쓸 수 있다.

---

## 문제: 트랜잭션 1만 개를 어떻게 요약하나

블록 하나에 트랜잭션이 10,000개 있다고 해보자. 이걸 해시 하나로 요약하려면 어떻게 해야 할까?

가장 단순한 방법은 트랜잭션 10,000개를 전부 이어붙여서 해시를 한 번에 계산하는 것이다. 해시 하나는 나온다. 그런데 문제가 있다.

나중에 누군가 "트랜잭션 #3,741번이 이 블록에 들어있나요?"라고 물어보면, 확인하기 위해 10,000개를 전부 가져와서 재계산해야 한다. 용량도 크고 시간도 오래 걸린다.

라이트 노드(스마트폰처럼 저장 공간이 작은 장치)는 블록 헤더만 저장하고 풀데이터는 안 갖고 있다. 이 라이트 노드가 특정 트랜잭션이 유효한지 어떻게 검증할까?

Merkle Tree는 이 문제를 우아하게 푼다.

---

## Merkle Tree 구조

아이디어는 간단하다. 트랜잭션들을 두 개씩 짝지어서 해시하고, 그 결과를 또 두 개씩 짝지어서 해시하고, 반복한다. 피라미드처럼 쌓아올리면 마지막에 단 하나의 해시가 남는다. 이걸 **Merkle Root**라고 부른다.

트랜잭션이 4개(A, B, C, D)인 예시:

```
            [Root]
           /       \
       [H(AB)]   [H(CD)]
       /    \    /    \
    [H(A)] [H(B)] [H(C)] [H(D)]
      A      B      C      D
```

1. 먼저 트랜잭션 각각을 해시한다: H(A), H(B), H(C), H(D)
2. 두 개씩 이어붙여서 다시 해시한다: H(AB) = hash(H(A) + H(B))
3. 한 번 더: Root = hash(H(AB) + H(CD))

블록 헤더에는 이 Root 값 하나만 저장된다.

---

## Merkle Proof: 특정 트랜잭션이 있다는 걸 증명하는 법

진짜 마법은 여기서 시작된다.

라이트 노드가 "트랜잭션 C가 이 블록에 있나요?"라고 풀노드에게 물었다고 하자. 풀노드는 트랜잭션 10,000개를 전부 보내는 대신, **증명 경로(Merkle Proof)**만 보내줄 수 있다.

트랜잭션이 4개인 위 예시에서 C를 증명하려면:

```
필요한 값: H(D), H(AB)
```

라이트 노드는 이렇게 검증한다:

1. C를 해시해서 H(C)를 구한다.
2. 받은 H(D)와 합쳐서 H(CD)를 구한다.
3. 받은 H(AB)와 합쳐서 Root를 구한다.
4. 자기가 갖고 있는 블록 헤더의 Merkle Root와 비교한다. 같으면 C는 이 블록에 있다.

전체 트랜잭션 데이터 없이 두 개의 해시값만으로 증명이 된다.

트랜잭션이 10,000개여도 증명에 필요한 해시 개수는 **log₂(10,000) ≈ 13개**다. 이게 `log(n)` 복잡도의 의미다.

---

## 왜 이게 중요한가

이더리움 블록 하나에는 세 개의 Merkle Root가 있다.

- **txRoot**: 이 블록에 포함된 트랜잭션 목록의 Merkle Root
- **receiptRoot**: 각 트랜잭션의 실행 결과(영수증) Merkle Root
- **stateRoot**: 이더리움 전체 계정 상태의 Root (다음 편에서 자세히)

이 세 개의 값이 블록 헤더에 들어간다. 블록 헤더는 고작 수백 바이트지만, 이 헤더 하나로 수기가바이트짜리 블록 데이터 전체의 무결성을 검증할 수 있다.

스마트폰처럼 저장 공간이 작은 라이트 노드가 블록체인과 상호작용할 수 있는 게 Merkle Tree 덕분이다. 블록 헤더만 저장하고, 필요할 때 Merkle Proof만 받아서 검증한다. 이 구조를 **SPV(Simplified Payment Verification)**라고 부른다.

---

## 트랜잭션 개수가 홀수면?

트랜잭션이 5개처럼 홀수일 때는 마지막 트랜잭션을 복사해서 짝수로 맞춘다.

```
[H(A)] [H(B)] [H(C)] [H(D)] [H(E)] [H(E)]  ← E가 복제됨
```

구현상의 편의를 위한 처리다. 비트코인과 이더리움 모두 이 방식을 쓴다.

---

## 이더리움이 쓰는 Patricia Merkle Trie

이더리움의 stateRoot는 일반 Merkle Tree가 아니라 **Patricia Merkle Trie**를 쓴다. 이유는 블록마다 잔액이 바뀌는 계정은 극소수인데, 그때마다 전체 트리를 재계산하면 너무 비효율적이기 때문이다.

Patricia Merkle Trie는 변경된 부분만 업데이트하는 구조다. 이건 다음다음 편에서 상태 트리를 다룰 때 자세히 살펴본다.
