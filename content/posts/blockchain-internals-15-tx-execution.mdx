지금까지 트랜잭션이 서명되고, 전파되고, mempool에 들어가고, 블록에 포함되는 과정을 봤다.

이제 블록에 담긴 트랜잭션이 EVM에서 어떻게 실제 실행되는지 따라가보자.

---

## 트랜잭션 실행 전: 유효성 검사

EVM이 트랜잭션을 실행하기 전에 먼저 기본 유효성 검사가 이루어진다.

```
1. 서명이 유효한가?
   → 서명에서 발신자 주소를 복원하고 확인

2. Nonce가 맞는가?
   → 온체인 nonce와 트랜잭션 nonce가 일치해야 함

3. 잔액이 충분한가?
   → balance >= gasLimit × maxFeePerGas + value
   (최악의 경우 가스를 다 쓰더라도 감당할 수 있어야 함)

4. Gas limit이 블록 가스 한도를 초과하지 않는가?
```

하나라도 실패하면 트랜잭션은 블록에 포함될 수 없다. 이런 트랜잭션은 mempool에서 걸러진다.

## 실행 흐름: 단계별 추적

Alice가 ERC-20 컨트랙트의 `transfer(Bob, 100)` 을 호출하는 트랜잭션을 예시로 따라간다.

### 1단계: 가스 선납

실행 전에 최대 가스 비용을 Alice 잔액에서 먼저 차감한다.

```
차감액 = gasLimit × effectiveGasPrice
Alice.balance -= gasLimit × effectiveGasPrice
```

실행이 끝나면 남은 가스를 환불한다.

왜 선납인가? 실행 중간에 "가스 부족"으로 멈추더라도, 이미 소모된 컴퓨팅 자원의 대가는 받아야 하기 때문이다.

### 2단계: 트랜잭션 타입 결정

`to` 주소가 있으면 → 메시지 콜 (컨트랙트 호출 or 단순 ETH 전송)

`to`가 없으면 → 컨트랙트 배포 (CREATE)

단순 ETH 전송이면 EVM 실행 없이 잔액만 바뀌고 끝난다. 가스 21,000 고정.

### 3단계: EVM 실행

ERC-20 컨트랙트의 `transfer` 바이트코드가 실행된다.

실행 컨텍스트:
```
msg.sender: Alice
msg.value:  0 (ETH 전송 없음)
calldata:   a9059cbb + Bob 주소 + 100 (ABI 인코딩)
```

EVM은 바이트코드를 한 옵코드씩 실행하면서 스택을 조작한다. 내부적으로 Alice 잔액 감소, Bob 잔액 증가, Transfer 이벤트 emit이 일어난다.

### 4단계: 가스 추적

옵코드가 실행될 때마다 해당 가스 비용이 차감된다.

남은 가스가 0이 되면 **Out of Gas** 예외가 발생한다. 현재까지의 모든 상태 변경이 롤백되고 실행이 종료된다.

### 5단계: 완료 또는 Revert

**정상 완료**: RETURN 옵코드로 종료. 상태 변경 확정. 남은 가스 환불.

**Revert**: 명시적 `require` 실패나 `revert()` 호출. 상태 변경 롤백. 소모된 가스는 환불 없음.

**Out of Gas**: 가스 소진. 상태 변경 롤백. 가스 전액 소모.

## 콜 스택: 컨트랙트가 컨트랙트를 부를 때

컨트랙트 A가 컨트랙트 B를 호출하면, EVM은 새 실행 컨텍스트를 만들어 스택에 쌓는다.

```
[A 실행 중]
  → B 호출 (CALL)
    [B 실행 중]
      → C 호출 (CALL)
        [C 실행 중]
        [C 완료]
      [B 계속 실행]
    [B 완료]
  [A 계속 실행]
```

이 콜 스택의 최대 깊이는 1024다. 재귀 호출이 1024번을 넘으면 스택 오버플로우로 revert된다.

각 CALL은 가스를 전달한다. B에게 전달하는 가스가 부족하면 B가 Out of Gas로 revert된다. 이때 A는 계속 실행될 수 있다 — CALL 명령어가 성공/실패 여부를 반환값으로 알려주기 때문에, A가 이를 체크하면 된다.

---

## Revert는 어디까지 롤백되나

```
A 실행
  → B 호출
    → B 내부에서 revert 발생!
  B의 상태 변경 전부 롤백
  → 하지만 A는 계속 실행될 수 있음
  A가 B의 실패를 확인하고 처리하면 A의 변경은 유지됨
```

A가 B의 실패를 확인하지 않으면? Solidity에서 `b.call(...)` 대신 `B.foo()`로 호출하면 B가 revert될 때 A도 자동으로 revert된다. `call()`로 저수준 호출하면 실패 여부를 수동으로 체크해야 한다.

## Receipt: 트랜잭션 실행의 영수증

트랜잭션이 완료되면 **Receipt(영수증)**가 생성된다. 블록의 receiptsRoot에 포함된다.

```json
{
  "transactionHash": "0x...",
  "blockNumber": "0x12f45a",
  "gasUsed": "0x5208",
  "status": "0x1",          // 1 = 성공, 0 = 실패
  "logs": [...]             // 이벤트 로그
}
```

`status: 0`이면 실패한 트랜잭션이다. 실패해도 블록에 포함되고 가스는 소모된다.

## 로그(Log): 이벤트가 기록되는 방식

Solidity의 `emit Transfer(from, to, amount)` 같은 이벤트는 EVM의 LOG 옵코드로 구현된다.

로그는 스토리지보다 훨씬 저렴하다. (LOG3 옵코드 기준 약 1,875 가스 vs SSTORE 20,000 가스)

로그는 Storage Trie에 저장되지 않는다. 따로 receipts 구조에 포함된다. 컨트랙트에서 직접 읽을 수 없다 (EVM 내부에서 과거 로그 조회 불가). 대신 Ethers.js의 `queryFilter()`나 The Graph 같은 인덱서를 통해 외부에서 조회한다.

이벤트의 첫 번째 topic은 이벤트 시그니처의 keccak256 해시다.

```
Transfer(address,address,uint256) 이벤트의 topic0
= keccak256("Transfer(address,address,uint256)")
= ddf252ad...
```

이 값으로 특정 이벤트만 필터링할 수 있다.
