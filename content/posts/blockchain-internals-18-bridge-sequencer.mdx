지금까지 롤업의 개념과 원리를 살펴봤다. 마지막 편에서는 실제 L2 인프라가 어떻게 돌아가는지, 시퀀서가 뭔지, 브릿지가 어떻게 작동하는지를 다룬다.

---

## 시퀀서(Sequencer)란

L2에서 트랜잭션을 처리하는 핵심 주체가 **시퀀서(Sequencer)**다.

사용자가 L2에 트랜잭션을 보내면, 시퀀서가 이를 받아서 순서를 정하고 실행한다. 이후 배치로 묶어서 L1에 게시한다.

```
사용자 트랜잭션
     ↓
[시퀀서]
  - 트랜잭션 수신 및 순서 결정
  - 즉각 실행 (L2 실시간 확인)
  - 주기적으로 L1에 배치 게시
     ↓
[L1 이더리움]
  - 데이터 보존
  - 최종 확정
```

시퀀서가 있으면 L2에서 거의 즉각적인 응답이 가능하다. L1 블록(12초)을 기다릴 필요 없이 시퀀서가 바로 실행 결과를 알려주기 때문이다.

---

## 시퀀서의 중앙화 리스크

문제가 있다. 현재 Arbitrum, Optimism, zkSync 등 대부분의 L2는 **단일 시퀀서**로 운영된다.

단일 시퀀서의 위험:

**검열(Censorship)**: 시퀀서가 특정 주소의 트랜잭션을 의도적으로 무시할 수 있다.

**다운타임**: 시퀀서 서버가 중단되면 L2 전체가 멈춘다. Arbitrum은 2023년에 약 70분간 다운된 적이 있다.

**MEV 독점**: 시퀀서가 트랜잭션 순서를 마음대로 정하므로 모든 MEV를 독점한다.

"하지만 이더리움은 탈중앙화 아닌가?" — 맞다. 그래서 이 중앙화가 문제로 지목된다. 모든 주요 L2팀이 시퀀서 탈중앙화를 로드맵에 올려두고 있지만, 실제 구현은 아직 더디다.

---

## 강제 트랜잭션 포함(Force Inclusion)

시퀀서가 검열해도 빠져나올 수 있는 탈출구가 있다.

**Force Inclusion** 또는 **Escape Hatch**라고 부른다.

Arbitrum을 예로 들면:

1. 시퀀서가 내 트랜잭션을 계속 무시한다면
2. L1의 SequencerInbox 컨트랙트에 직접 트랜잭션을 제출한다
3. 시퀀서가 일정 기간(24시간) 내에 이를 포함하지 않으면
4. 누구나 강제로 이 트랜잭션을 L2에 포함시킬 수 있다

느리지만, 시퀀서가 영원히 검열할 수 없다. 최악의 경우에도 24시간 후에는 트랜잭션이 실행된다.

이 메커니즘이 L2의 탈중앙화 보장의 핵심이다. 시퀀서를 믿지 않아도 자산을 L1으로 되돌릴 수 있다.

## L1 ↔ L2 자산 이동

### L1 → L2 (입금, Deposit)

```
1. L1의 브릿지 컨트랙트에 ETH/토큰을 전송
2. L2 측이 이를 감지하고 L2 주소에 상응하는 자산 발행
3. 보통 수 분 내 완료
```

입금은 빠르다. L1에서 L2로 "자산이 잠겼다"는 증거를 L2가 확인하면 되기 때문이다.

### L2 → L1 (출금, Withdrawal)

Optimistic Rollup 기준:

```
1. L2에서 출금 요청
2. L1에 출금 요청 기록 (state root 갱신)
3. 7일 이의 제기 기간 대기
4. 기간 완료 후 L1 브릿지에서 자산 수령
```

ZK Rollup 기준:

```
1. L2에서 출금 요청
2. ZK proof 생성 (수분~수시간)
3. proof와 함께 L1에 출금 요청
4. L1이 proof 검증 후 자산 즉시 지급
```

### 써드파티 브릿지의 등장

7일 대기가 불편해서 등장한 게 **써드파티 브릿지**다. Hop Protocol, Stargate, Across 같은 서비스다.

```
사용자 → (Optimistic L2에서 즉시 출금 요청) → 브릿지
브릿지 → (L1에서 자산 즉시 지급)
브릿지 → (7일 후 L1에서 정식 출금으로 자산 회수)
```

브릿지가 7일을 기다리는 대신, 사용자는 소정의 수수료를 내고 즉시 자산을 받는다.

단, 써드파티 브릿지는 별도 스마트 컨트랙트 위험이 있다. 브릿지 컨트랙트 해킹으로 수억 달러가 도난당한 사례가 여러 번 있었다. (Ronin 브릿지 6억 달러, Wormhole 3.2억 달러 등)

## EIP-4844: Blob 트랜잭션으로 L2 수수료가 낮아진 이유

2024년 3월 Cancun 업그레이드로 도입된 EIP-4844의 핵심인 **Blob(Binary Large Object)** 트랜잭션이다.

이전에는 L2가 트랜잭션 데이터를 L1 calldata에 올렸다. Calldata는:
- 블록에 영구적으로 저장된다
- 모든 풀 노드가 평생 보관해야 한다
- 따라서 비쌌다

Blob은 다르다:
- 블록에 첨부되는 별도 데이터 공간 (블록당 최대 6개, 각 128KB)
- 약 18일 후 자동 삭제된다
- 검증에 필요한 해시(KZG commitment)만 영구 보존
- 별도의 저렴한 가스 요금 체계 (blob gas)

L2 입장에서 calldata 대신 Blob을 쓰면 비용이 10분의 1 수준으로 줄어든다. Cancun 업그레이드 이후 Arbitrum과 Optimism의 트랜잭션 수수료가 $0.001 미만으로 떨어졌다.

## Danksharding: 앞으로 어떻게 더 확장될 예정인가

EIP-4844는 **Proto-Danksharding**이라는 별칭답게, 완전한 Danksharding의 준비 단계다.

완전한 Danksharding의 목표:
- 블록당 Blob 수를 현재 6개에서 최대 64개로 확장
- **데이터 가용성 샘플링(DAS)**: 라이트 노드도 무작위 샘플링으로 전체 Blob 데이터의 가용성을 검증할 수 있는 기술
- 이론상 L2 처리량을 현재 대비 10배 이상 확장

DAS가 핵심이다. 지금은 Blob 데이터를 검증하려면 풀 노드가 전체를 다운받아야 한다. DAS를 쓰면 라이트 노드도 일부만 다운받아서 전체 가용성을 통계적으로 검증할 수 있다. 탈중앙화를 유지하면서 확장성을 높이는 방법이다.
