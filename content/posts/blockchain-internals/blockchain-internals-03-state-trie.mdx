지난 편에서 블록 헤더를 뜯어봤다. 그 안에 `stateRoot`라는 필드가 있었다.

> "블록 실행 후 이더리움 전체 계정 상태를 요약한 해시값"

이번 편에서는 이게 실제로 어떻게 생겼는지 들어가 본다.

---

## 비트코인과 이더리움이 잔액을 저장하는 방식이 다르다

먼저 배경 지식 하나. 블록체인이 잔액을 저장하는 방식은 크게 두 가지다.

### UTXO 모델 (비트코인)

UTXO는 Unspent Transaction Output, "사용하지 않은 거래 출력값"이라는 뜻이다. 개념이 좀 낯선데, 은행 계좌가 아니라 **지폐**로 생각하면 된다.

내 지갑에 만원짜리 3장이 있다고 해보자 (= UTXO가 3개). 5천원을 내려면 만원짜리를 꺼내서 5천원짜리 거스름돈을 받는다. 잔액이라는 개념이 없고, 내가 쓸 수 있는 조각들만 있다.

비트코인이 이 방식을 쓴다. "내 잔액"이 어딘가에 숫자로 저장된 게 아니라, 내 주소로 들어왔지만 아직 안 쓴 트랜잭션 출력값들의 합이 잔액이다.

### Account 모델 (이더리움)

이더리움은 우리가 아는 은행 계좌 방식이다. 각 주소마다 계정이 있고, 거기에 잔액이 숫자로 저장된다.

```
0xAbC...: { balance: 5 ETH, nonce: 12 }
0xDeF...: { balance: 0.3 ETH, nonce: 1 }
```

직관적이고 스마트 컨트랙트 구현이 쉽다. 컨트랙트가 "이 계정 잔액이 얼마야?"를 바로 조회할 수 있으니까. UTXO 방식으로는 스마트 컨트랙트 짜기가 훨씬 복잡해진다.

---

## 문제: 계정이 수억 개인데 어떻게 저장하나

Account 모델의 약점은 "전체 계정 상태"를 어딘가에 다 저장해야 한다는 거다.

이더리움 메인넷의 활성 계정 수는 수억 개를 넘는다. 이걸 단순한 테이블로 저장하면:

- 계정 하나를 수정하면 전체 테이블이 바뀐다
- 블록마다 `stateRoot` (상태 요약 해시)를 새로 계산하려면 전체를 다시 해싱해야 한다
- 매우 느리고 비효율적이다

그래서 이더리움은 **Patricia Merkle Trie**라는 자료구조를 쓴다.

---

## Trie가 뭔지부터

Trie는 문자열을 저장하는 트리 자료구조다. 각 노드가 문자 하나씩을 담고, 루트부터 특정 노드까지의 경로가 키(key)가 된다.

예를 들어 "apple", "app", "banana"를 Trie에 저장하면:

```
root
├── a
│   └── p
│       └── p → "app" 저장
│           └── l
│               └── e → "apple" 저장
└── b
    └── a
        └── n
            └── a
                └── n
                    └── a → "banana" 저장
```

공통 접두어("app")를 공유해서 공간을 아끼고, 검색이 빠르다.

이더리움은 이걸 이더리움 계정 주소로 응용한다. 주소(0xABCDEF...)가 키고, 계정 상태가 값이다.

---

## Patricia Merkle Trie: 세 가지를 합친 것

이더리움의 State Trie는 세 가지 개념을 합친 거다.

- **Trie**: 주소로 계정을 빠르게 조회
- **Patricia**: 경로를 압축해서 공간 절약 (공통 접두어가 긴 경우 중간 노드를 압축)
- **Merkle**: 각 노드를 해시값으로 참조 → 루트 해시 하나로 전체 상태를 요약

마지막 "Merkle" 성질이 핵심이다. 어느 계정의 잔액이 1이라도 바뀌면, 그 계정에서 루트까지 경로에 있는 모든 노드의 해시가 바뀐다. 그래서 `stateRoot`만 비교해도 "두 상태가 같은지"를 즉시 알 수 있다.

---

## 이더리움 계정에는 두 종류가 있다

State Trie에 담기는 계정은 두 가지다.

### EOA (Externally Owned Account, 외부 소유 계정)

MetaMask처럼 개인키로 제어하는 일반 지갑이다.

```
{
  nonce:   12,        // 이 주소에서 보낸 트랜잭션 수
  balance: 5000000000000000000  // 5 ETH (wei 단위)
}
```

### CA (Contract Account, 컨트랙트 계정)

스마트 컨트랙트 배포 주소다. 개인키가 없고, 코드에 의해서만 제어된다.

```
{
  nonce:       1,
  balance:     0,
  codeHash:    0x5a3f...,  // 컨트랙트 바이트코드의 해시
  storageRoot: 0x8b2e...,  // 이 컨트랙트의 스토리지 트리 루트
}
```

EOA에 없는 두 필드가 있다.

- `codeHash`: 컨트랙트 바이트코드를 해시한 값. 실제 코드는 별도로 저장하고, 여기엔 해시만 담는다.
- `storageRoot`: 이 컨트랙트가 저장한 데이터(mapping, array 등)를 담은 또 다른 Merkle Trie의 루트. 컨트랙트마다 독립적인 스토리지 공간이 있다.

---

## 트랜잭션 하나가 실행되면 어떻게 되나

Alice가 Bob한테 1 ETH를 보내는 트랜잭션을 예로 들면:

```
트랜잭션 실행 전:
  Alice: { balance: 5 ETH, nonce: 12 }
  Bob:   { balance: 2 ETH, nonce: 0 }

트랜잭션 실행 후:
  Alice: { balance: 3.9997 ETH, nonce: 13 }  ← 1 ETH 빠지고, 가스비 빠지고, nonce 증가
  Bob:   { balance: 3 ETH, nonce: 0 }        ← 1 ETH 추가
```

이 변화가 State Trie에 반영되면, Alice와 Bob 노드에서 루트까지 경로에 있는 모든 노드의 해시가 다시 계산된다. 그 결과로 새로운 `stateRoot`가 만들어지고, 이게 블록 헤더에 기록된다.

블록 하나에 트랜잭션이 200개 있으면, 200번의 상태 변경이 순서대로 일어나고, 최종 `stateRoot`가 헤더에 들어간다.

---

## nonce가 뭔지도 짚고 가자

계정 필드에 `nonce`라는 게 있는데, 입문 시리즈에서 PoW의 nonce(채굴용 숫자)가 나왔던 것과 이름이 같아서 헷갈릴 수 있다. 전혀 다른 거다.

계정의 `nonce`는 **이 주소에서 보낸 트랜잭션의 총 개수**다.

왜 필요하냐면, 같은 트랜잭션을 두 번 실행하는 **재전송 공격(Replay Attack)**을 막기 위해서다. 트랜잭션에는 반드시 `nonce`가 포함되는데, 이더리움 노드는 `현재 계정 nonce == 트랜잭션 nonce`인 것만 실행한다. Alice의 nonce가 12라면, 다음 트랜잭션은 반드시 nonce 13이어야 한다.

"트랜잭션이 왜 순서대로 처리되냐"는 것도 이 nonce 때문이다.

---

## 세 개의 Trie

블록 헤더에 Root가 세 개 있다고 했는데, 세 가지 Trie가 각각 다른 걸 저장한다.

| Trie | Root 필드 | 저장하는 것 |
|------|-----------|------------|
| State Trie | `stateRoot` | 모든 계정의 현재 상태 |
| Transaction Trie | `transactionsRoot` | 이 블록의 트랜잭션 목록 |
| Receipt Trie | `receiptsRoot` | 각 트랜잭션의 실행 결과 |

State Trie는 블록마다 누적해서 업데이트된다. 현재 이더리움의 "전체 상태"를 갖고 있다. Transaction Trie와 Receipt Trie는 해당 블록에만 해당하는 순간 스냅샷이다.

## 용어 정리

**UTXO (Unspent Transaction Output)**
비트코인이 잔액을 표현하는 방식. "잔액"이라는 개념 없이, 사용하지 않은 거래 출력값의 합이 잔액이다.

**Account 모델**
이더리움이 채택한 방식. 계정마다 balance, nonce 등을 직접 저장한다. 직관적이고 스마트 컨트랙트에 적합하다.

**Patricia Merkle Trie**
이더리움이 State를 저장하는 자료구조. Trie의 빠른 조회 + Patricia의 공간 압축 + Merkle의 해시 검증을 합친 구조.

**nonce (계정)**
해당 주소에서 보낸 트랜잭션의 누적 카운트. 재전송 공격을 막고 트랜잭션 순서를 강제한다. PoW의 nonce와 이름만 같고 전혀 다른 개념이다.

**codeHash / storageRoot**
컨트랙트 계정에만 있는 필드. `codeHash`는 컨트랙트 바이트코드의 해시, `storageRoot`는 컨트랙트 내부 데이터를 담은 별도 Trie의 루트다.
