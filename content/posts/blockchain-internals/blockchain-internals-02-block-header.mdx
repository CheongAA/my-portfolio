지난 편에서 Merkle Tree를 배웠다. 트랜잭션을 2개씩 짝지어서 해시하고, 맨 꼭대기에 Merkle Root를 만드는 구조.

그 Merkle Root가 실제로 어디에 저장되는지 궁금하지 않았는가.

답은 **블록 헤더**다. 이번 편에서는 블록 헤더를 직접 뜯어본다.

---

## 블록은 두 부분으로 나뉜다

흔히 "블록"을 하나의 덩어리로 생각하는데, 실제로는 두 파트로 나뉜다.

```
[ 블록 헤더 ]  ← 블록을 요약한 메타데이터
[ 트랜잭션 목록 ]  ← 실제 거래 내역들
```

헤더는 블록의 "신분증" 같은 것이다. 내용 전체를 담지는 않지만, 그 블록을 검증하는 데 필요한 모든 정보가 담겨 있다.

Etherscan에서 아무 블록이나 클릭하면 나오는 낯선 필드들이 다 헤더에 있는 값들이다.

---

## 헤더 필드 하나씩 뜯어보기

이더리움 블록 헤더에는 15개 이상의 필드가 있다. 중요한 것들만 골라서 보자.

### parentHash

```
parentHash: 0x3a4e9f...
```

이전 블록의 해시값이다. 이게 "체인"의 실체다. 모든 블록이 이전 블록의 해시를 담고 있기 때문에, 중간 블록 하나를 바꾸면 그 이후 블록들의 `parentHash`가 전부 틀어진다.

블록체인의 불변성이 이 필드 하나에서 나온다.

### stateRoot

```
stateRoot: 0xd7c2a1...
```

이 블록이 처리된 직후, **이더리움 전체 계정의 상태를 담은 트리의 루트 해시**다. 모든 계정의 잔액, 컨트랙트 코드, 스토리지가 여기 요약돼 있다.

블록이 실행되기 전과 후의 `stateRoot`가 다르다. 트랜잭션이 누군가의 잔액을 바꿨으니까. 이게 어떻게 동작하는지는 다음 편인 상태 트리에서 자세히 다룬다.

### transactionsRoot

```
transactionsRoot: 0xa3f9c2...
```

지난 편에서 배운 **Merkle Root**다. 이 블록에 담긴 모든 트랜잭션을 Merkle Tree로 쌓아서 나온 루트 해시. 트랜잭션 하나라도 바뀌면 이 값이 달라진다.

### receiptsRoot

```
receiptsRoot: 0x2b8f6e...
```

트랜잭션마다 실행 결과(영수증)가 하나씩 생긴다. 성공/실패 여부, 가스 사용량, 이벤트 로그 등이 담긴다. 그 영수증들도 Merkle Tree로 묶어서 루트 해시를 만든 게 이 필드다.

`transactionsRoot`가 "어떤 거래를 했는지"라면, `receiptsRoot`는 "그 거래가 어떻게 됐는지"다.

### number

```
number: 21000000
```

이 블록의 높이. 제네시스 블록이 0번이고, 그 위에 쌓일 때마다 1씩 증가한다. "블록 #21,000,000" 할 때의 그 숫자다.

### timestamp

```
timestamp: 1741574400
```

블록이 생성된 시각. 유닉스 타임스탬프(1970년 1월 1일 기준 초 단위)로 표시된다.

흥미로운 점은 이더리움 PoS에서는 블록이 정확히 12초마다 생성되도록 설계돼 있다는 거다. 그래서 `number`와 `timestamp`를 알면 대략 언제 블록이 생성됐는지 계산이 된다.

### gasLimit / gasUsed

```
gasLimit: 30000000
gasUsed:  14523781
```

`gasLimit`은 이 블록에서 쓸 수 있는 최대 가스 총량이다. 이걸 초과하는 트랜잭션은 블록에 못 들어간다.

`gasUsed`는 실제로 사용된 가스. 이 블록 안의 모든 트랜잭션이 쓴 가스를 합친 값이다.

가스는 다음 묶음인 EVM 파트에서 자세히 다루는데, 지금은 "블록마다 처리할 수 있는 연산량의 한계가 있다"는 것만 알면 충분하다.

### baseFeePerGas

```
baseFeePerGas: 12 Gwei
```

EIP-1559(2021년 8월)로 도입된 필드다. 이 블록의 거래 최소 수수료. 블록이 가득 찰수록 올라가고, 여유 있으면 내려간다. 이 금액은 채굴자(검증자)한테 가는 게 아니라 **소각(burn)**된다.

"왜 수수료를 태워버리지?"라는 의문이 생길 수 있는데, 이건 11편 가스 편에서 제대로 풀어낸다.

### extraData

```
extraData: 0x
```

블록 생성자가 자유롭게 넣을 수 있는 필드다. 채굴자들이 여기에 자기 소속 풀 이름이나 메시지를 넣기도 했다. PoS로 넘어온 이후로는 잘 쓰이지 않는다.

## 세 개의 Root가 담는 것

정리해보면 블록 헤더에는 Merkle Root가 세 개 들어 있다.

| 필드 | 담는 것 |
|------|---------|
| `transactionsRoot` | 이 블록의 트랜잭션 목록 |
| `receiptsRoot` | 각 트랜잭션의 실행 결과 |
| `stateRoot` | 블록 실행 후 전체 계정 상태 |

이 세 개의 Root가 있으면, 풀노드는 블록 전체를 갖지 않고도 특정 트랜잭션이 있는지, 특정 계정 잔액이 얼마인지를 증명할 수 있다.

---

## 블록 해시가 불변성을 만드는 방법

앞서 본 필드들을 전부 합쳐서 해시하면 이 블록의 고유 식별값인 **블록 해시**가 된다.

```
블록 해시 = hash(parentHash + stateRoot + transactionsRoot + ... 모든 헤더 필드)
```

그리고 이 블록 해시는 다음 블록의 `parentHash`로 들어간다.

```
블록 N의 해시 → 블록 N+1의 parentHash
블록 N+1의 해시 → 블록 N+2의 parentHash
```

누군가 블록 N의 `gasUsed`를 단 1바이트라도 바꾸면, 블록 N의 해시가 달라진다. 그러면 블록 N+1의 `parentHash`와 안 맞는다. 블록 N+1부터 끝까지 전부 다시 계산해야 한다. 그것도 수천 개의 노드에서 동시에.

불변성은 이 연쇄 구조에서 나온다.

## 직접 확인해보기

Etherscan에서 확인하는 방법:

1. [etherscan.io](https://etherscan.io) 접속
2. 상단 검색창에 블록 번호 입력 (예: `21000000`)
3. **"More Details"** 클릭하면 숨겨진 필드들이 나온다
4. `State Root`, `Transactions Root`, `Receipts Root` 값 확인

또는 Ethers.js로 직접 가져올 수도 있다.

```js
import { ethers } from "ethers";

const provider = new ethers.JsonRpcProvider("https://rpc.ankr.com/eth");
const block = await provider.getBlock(21000000);

console.log(block.parentHash);          // 이전 블록 해시
console.log(block.transactionsRoot);    // 트랜잭션 Merkle Root
console.log(block.stateRoot);           // 상태 트리 루트
console.log(block.gasLimit);
console.log(block.baseFeePerGas);
```

## 용어 정리

**블록 헤더 (Block Header)**
블록의 메타데이터. 트랜잭션 목록은 포함하지 않지만, 검증에 필요한 모든 요약 정보가 담겨 있다.

**stateRoot**
블록 실행 후 이더리움 전체 계정 상태를 Merkle Tree로 요약한 해시값. 다음 편에서 이 트리가 어떻게 생겼는지 자세히 본다.

**baseFeePerGas**
EIP-1559로 도입된 블록 기본 수수료. 자동으로 조정되며, 검증자가 아닌 프로토콜로 소각된다.

---

다음 편에서는 `stateRoot`가 가리키는 **상태 트리(State Trie)**를 파헤친다. 이더리움이 수억 개의 계정 잔액을 어떻게 저장하고, 트랜잭션 하나가 실행될 때 그 구조가 어떻게 업데이트되는지를 다룬다.
