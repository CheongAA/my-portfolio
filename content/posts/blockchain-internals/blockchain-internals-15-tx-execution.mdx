Part 4의 마지막 편이다.

P2P로 퍼진 트랜잭션이 mempool에 대기하다가 블록에 담긴다(14편 MEV에서 이 과정도 봤다). 블록에 담긴 트랜잭션은 이제 EVM이 실행한다.

이번 편은 그 실행 과정 전체를 처음부터 끝까지 추적한다. 8~11편에서 배운 EVM, 스토리지, 가스가 모두 연결되는 편이다.

---

## 전체 흐름 한눈에

```
트랜잭션 수신
    ↓
1. 유효성 검사 (nonce, 서명, 잔액)
    ↓
2. 가스 선납 (gas limit × gas price만큼 잔액 차감)
    ↓
3. EVM 실행
   - 컨트랙트 호출이면: 바이트코드 실행
   - 단순 ETH 전송이면: 21,000 가스 차감 후 완료
    ↓
4. 가스 정산 (남은 가스 환불, baseFee 소각, tip 지급)
    ↓
5. 상태 변경 반영 (stateRoot 업데이트)
    ↓
6. 영수증(Receipt) 생성 및 로그 기록
```

각 단계를 들여다보자.

---

## 1단계: 유효성 검사

EVM이 실행하기 전에 먼저 트랜잭션이 유효한지 확인한다.

```
- 서명이 올바른가? (보낸 사람이 정말 이 트랜잭션에 서명했는가)
- nonce가 맞는가? (계정의 현재 nonce + 1인가)
- 잔액이 충분한가? (gas limit × gas price + value ≤ 잔액)
- gas limit이 최소값(21,000) 이상인가?
```

하나라도 실패하면 트랜잭션은 처리되지 않는다. 가스도 소모되지 않는다.

---

## 2단계: 가스 선납

유효성 검사를 통과하면, `gas limit × gas price`만큼 송신자 잔액에서 먼저 차감한다.

```
gas limit: 100,000
gas price: 20 gwei

선납금 = 100,000 × 20 gwei = 0.002 ETH
→ 송신자 잔액에서 0.002 ETH 즉시 차감
```

실제로 다 쓰지 않으면 남은 금액은 나중에 환불된다.

선납하는 이유: EVM 실행 도중 "잔액 부족"으로 중단되는 상황을 방지한다. 실행 전에 가스비 전액을 잠가두는 것이다.

---

## 3단계: EVM 실행

### 단순 ETH 전송 (to 주소가 EOA)

바이트코드 실행 없이 21,000 가스를 소비하고 수신자 잔액을 증가시킨다.

### 컨트랙트 호출 (to 주소가 CA)

calldata를 입력으로 해당 컨트랙트의 바이트코드를 실행한다.

```
EVM 실행 컨텍스트:
- msg.sender: 트랜잭션 발신자
- msg.value: 보낸 ETH 금액
- calldata: 트랜잭션 data 필드
- gas: 남은 가스 (실행 중 계속 감소)
```

옵코드를 하나씩 실행할 때마다 해당 옵코드의 가스가 차감된다. 가스가 0이 되면 Out of Gas 에러가 발생한다.

---

## 콜 스택 — 컨트랙트가 컨트랙트를 호출할 때

컨트랙트 A가 컨트랙트 B를 호출하면 새로운 실행 컨텍스트가 생성되고 스택에 쌓인다.

```
A.execute() 호출
  └─ B.transfer() 호출 (CALL 옵코드)
       └─ C.approve() 호출 (CALL 옵코드)
```

```
콜 스택:
  [C.approve 실행 중]    ← 현재 실행
  [B.transfer 대기]
  [A.execute 대기]
```

C가 완료되면 B로 결과가 반환되고, B가 완료되면 A로 반환된다.

EVM의 콜 스택 최대 깊이는 **1024**다. 이를 초과하면 스택 오버플로우로 트랜잭션이 revert된다. 재귀 호출로 악의적으로 콜 스택을 채우는 공격("depth limit attack")을 방지하기 위해 만들어진 제한이다.

---

## revert — 상태 롤백

실행 중에 실패가 발생하면(Out of Gas, require 실패, assert 실패 등) **revert**가 발생한다.

중요한 것: revert는 **상태 변경을 되돌린다.** 트랜잭션이 시작된 이후의 모든 스토리지 변경, ETH 이동이 없던 일이 된다.

단, **가스는 환불되지 않는다.** revert까지 소비된 가스는 그대로 검증자에게 돌아간다.

```
트랜잭션이 50,000 가스를 쓰다가 revert 발생
→ 상태 변경: 전부 롤백 (없던 일)
→ 가스 50,000 gwei: 환불 안 됨 (검증자 수익)
```

컨트랙트가 다른 컨트랙트를 CALL로 호출했을 때 피호출자가 revert하면 어떻게 될까?

기본적으로 revert는 상위 호출로 전파된다. 하지만 `try/catch` (Solidity 0.6+)를 사용하면 특정 호출의 revert를 잡아서 처리할 수 있다.

---

## 4단계: 가스 정산

EVM 실행이 끝나면 가스를 정산한다.

```
선납금: 100,000 × 20 gwei = 0.002 ETH
실제 사용: 63,000 가스

남은 가스: 37,000 × 20 gwei = 0.00074 ETH → 송신자에게 환불
baseFee 소각: 63,000 × baseFee(예: 15 gwei) = 0.000945 ETH → 소각
tip 지급: 63,000 × priorityFee(예: 5 gwei) = 0.000315 ETH → 검증자
```

---

## 5단계: 상태 반영

트랜잭션 실행으로 변경된 상태(잔액, 컨트랙트 스토리지 등)가 이더리움 상태 트리에 반영된다.

3편에서 본 Patricia Merkle Trie의 루트 해시(stateRoot)가 업데이트된다.

블록 안의 모든 트랜잭션이 순서대로 실행된 후 최종 stateRoot가 블록 헤더에 기록된다.

---

## 6단계: Receipt와 로그

트랜잭션 실행 결과로 **영수증(Receipt)**이 생성된다.

```
Receipt 내용:
- transactionHash: 트랜잭션 식별자
- status: 1(성공) / 0(revert)
- gasUsed: 실제 소비된 가스
- logs: emit된 이벤트들
- blockNumber: 포함된 블록 번호
```

Solidity의 `emit Event(...)` 호출이 실행되면 그 데이터가 Receipt의 `logs` 배열에 기록된다.

이벤트 로그는 스토리지에 저장되지 않는다. Receipt에만 있다. 컨트랙트가 이벤트를 직접 읽을 수 없는 이유가 여기 있다. 이벤트는 외부 인덱서(Ethers.js의 `queryFilter`, The Graph 등)가 읽기 위한 것이다.

---

## 실습: 트랜잭션 Receipt 직접 확인

Ethers.js로 트랜잭션 영수증을 가져올 수 있다.

```javascript
const receipt = await provider.getTransactionReceipt(txHash);

console.log(receipt.status);     // 1 (성공) or 0 (실패)
console.log(receipt.gasUsed);    // BigInt: 실제 소비 가스
console.log(receipt.logs);       // 이벤트 로그 배열
```

Etherscan에서도 트랜잭션 해시로 검색하면 "Logs" 탭에서 모든 이벤트를 확인할 수 있다.

## 용어 정리

**가스 선납**
EVM 실행 전에 gas limit × gas price만큼 잔액에서 미리 차감하는 것. 실행 후 남은 가스는 환불된다.

**콜 스택**
컨트랙트가 컨트랙트를 호출할 때 쌓이는 실행 컨텍스트 스택. 최대 깊이 1024.

**revert**
트랜잭션 실행 실패 시 발생. 상태 변경은 롤백되지만 소비된 가스는 환불되지 않는다.

**Receipt (영수증)**
트랜잭션 실행 결과 기록. 성공/실패 여부, 가스 사용량, 이벤트 로그를 포함한다.
