Part 5는 이더리움의 한계와 그 해결책이다.

이더리움 메인넷은 바쁜 날 가스비가 수만 원을 넘는다. 단순한 토큰 전송에. 왜 이 문제가 생기는지, 왜 쉽게 고치지 못하는지를 이해하면 Layer 2가 왜 등장했는지 자연스럽게 이해된다.

---

## 이더리움의 TPS 한계

이더리움 메인넷의 처리량은 초당 약 15~30건(TPS)이다.

Visa는 초당 수천~수만 건을 처리한다. 모바일 결제 인프라 수준으로 가려면 이더리움이 수백~수천 배 더 빨라져야 한다.

왜 이 수준에 묶여 있을까?

---

## 구조적 이유: 모든 노드가 모든 걸 한다

이더리움의 각 풀 노드는 **모든 트랜잭션을 직접 실행하고 검증한다.**

블록에 트랜잭션 100건이 담기면, 전 세계 수천 개 노드가 각자 그 100건을 전부 실행한다. 이건 중복이다. 수천 배의 중복 연산.

중앙화된 시스템에서는 서버 한 대가 처리하면 끝이다. 블록체인은 모든 노드가 검증해야 하는 구조라 이 중복이 피할 수 없다.

**블록 생성 주기**와 **블록 가스 한도**가 TPS를 결정한다.

```
블록 생성 주기: 12초
블록 가스 한도: ~30,000,000 가스
ETH 전송 1건: 21,000 가스
→ 블록당 최대 ~1,428건
→ TPS = 1,428 / 12 ≈ 119 TPS (단순 전송 기준)

컨트랙트 호출이 섞이면: 실제 15~30 TPS
```

블록 가스 한도를 올리면 TPS가 늘어나지 않을까?

---

## 확장성 트릴레마

블록체인에서 세 가지 성질을 동시에 갖기 어렵다는 개념이다.

```
          탈중앙화
         /        \
        /          \
       /            \
      보안 --------- 확장성
```

**탈중앙화 (Decentralization)**
누구나 노드를 운영할 수 있어야 한다. 고사양 장비가 필요하지 않아야 한다.

**보안 (Security)**
공격자가 네트워크를 조작하기 어려워야 한다.

**확장성 (Scalability)**
많은 트랜잭션을 빠르고 저렴하게 처리할 수 있어야 한다.

블록 가스 한도를 10배 올리면 블록 크기가 커진다. 블록을 처리하고 검증하는 데 더 좋은 컴퓨터가 필요해진다. 일반 사용자가 노드를 운영하기 어려워지면서 탈중앙화가 약해진다.

비트코인이 오랫동안 블록 크기 논쟁(block size war)을 겪었던 이유가 이것이다. 크기를 키우면 확장성은 좋아지지만 탈중앙화가 약해진다.

이더리움은 탈중앙화와 보안을 유지하면서 확장성을 희생하는 쪽을 선택했다. L1 자체를 빠르게 하는 대신, L2를 통해 확장성 문제를 우회한다.

---

## 롤업의 핵심 아이디어

**롤업(Rollup)**은 이더리움 L1 위에서 동작하는 별도 체인이다.

핵심 아이디어는 간단하다.

> 연산은 L2에서 하고, L1은 데이터와 보안만 담당한다.

```
L2에서:
- 수천 건의 트랜잭션을 처리
- 결과를 하나의 배치로 압축

L1(이더리움 메인넷)에:
- 압축된 배치 데이터 제출
- 배치 데이터의 유효성 증명 (또는 이의 제기 기간) 제출
```

L2 사용자들은 L2 수수료만 낸다. L2 수수료는 L1보다 훨씬 싸다. 수천 건의 트랜잭션이 L1 비용을 나눠 부담하기 때문이다.

---

## 데이터 가용성 (Data Availability)

롤업이 L1 보안을 활용하려면 배치 데이터를 L1에 올려야 한다.

왜? 누구나 배치 데이터를 보고 검증할 수 있어야 하기 때문이다.

데이터를 외부 서버에만 올리면 서버가 데이터를 숨기거나 조작할 수 있다. 이더리움 L1에 올리면 누구나 언제든 확인할 수 있다.

이걸 **데이터 가용성(Data Availability)**이라고 한다. 롤업의 보안이 데이터 가용성에 의존한다.

EIP-4844(Proto-Danksharding)는 이 데이터를 더 저렴하게 L1에 올릴 수 있도록 "블롭(Blob)"이라는 임시 저장 공간을 도입했다. 이 업그레이드로 L2 수수료가 크게 하락했다.

---

## 보안을 L1에 위임한다는 것

롤업이 "L1 보안을 활용한다"는 게 구체적으로 무엇일까?

만약 L2 운영자가 잘못된 상태를 L1에 제출하면? 사용자가 잘못 처리된 자금을 L1 컨트랙트를 통해 되돌릴 수 있어야 한다.

이 보장이 어떻게 이루어지는지가 롤업의 두 가지 접근법(Optimistic vs ZK)을 나눈다. 다음 편에서 다룬다.

---

## 용어 정리

**확장성 트릴레마**
탈중앙화, 보안, 확장성을 동시에 완전히 달성할 수 없다는 블록체인 설계 원칙.

**롤업 (Rollup)**
L2에서 트랜잭션을 처리하고 압축된 결과를 L1에 제출하는 방식의 확장 솔루션.

**데이터 가용성 (Data Availability)**
배치 데이터가 누구나 접근 가능한 곳에 있어야 한다는 요구사항. 롤업 보안의 핵심 전제.

**EIP-4844 (Proto-Danksharding)**
L2가 L1에 데이터를 올리는 비용을 줄이기 위해 임시 블롭 저장소를 도입한 이더리움 업그레이드.
