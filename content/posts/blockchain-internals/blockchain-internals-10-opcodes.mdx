8편에서 Solidity 코드가 바이트코드로 컴파일된다는 걸 봤다. 9편에서 calldata 인코딩을 봤다.

이번엔 그 바이트코드가 실제로 어떻게 실행되는지 — **옵코드(Opcode)** 수준으로 내려간다.

---

## 옵코드란

옵코드는 EVM이 이해하는 기계어 명령어다. 각 옵코드는 1바이트로 표현되며, 256개(0x00 ~ 0xFF)가 정의돼 있다.

바이트코드는 이 옵코드들의 나열이다.

```
60 80 60 40 52 ...
↓
PUSH1 0x80
PUSH1 0x40
MSTORE
...
```

EVM은 이 옵코드들을 왼쪽에서 오른쪽으로 하나씩 읽으며 실행한다.

---

## 기본 옵코드들

### PUSH — 스택에 값 올리기

`PUSH1`부터 `PUSH32`까지 있다. 뒤따르는 1~32바이트를 스택에 올린다.

```
PUSH1 0x05   // 스택: [5]
PUSH1 0x03   // 스택: [5, 3]
```

### POP — 스택에서 꺼내 버리기

스택 맨 위 값을 꺼내서 버린다.

```
// 스택: [5, 3]
POP          // 스택: [5]
```

### ADD, MUL, SUB, DIV — 산술 연산

스택 맨 위 두 값을 꺼내 연산하고, 결과를 다시 올린다.

```
PUSH1 0x05   // 스택: [5]
PUSH1 0x03   // 스택: [5, 3]
ADD          // 3 + 5 → 스택: [8]
```

### MSTORE / MLOAD — 메모리 읽기/쓰기

```
PUSH1 0x42   // 값: 0x42
PUSH1 0x00   // 오프셋: 0
MSTORE       // memory[0] = 0x42
```

### SSTORE / SLOAD — 스토리지 읽기/쓰기

컨트랙트의 영구 저장소에 접근한다. 가장 비싼 옵코드 중 하나.

```
PUSH1 0x01   // 값: 1
PUSH1 0x00   // 슬롯: 0
SSTORE       // storage[0] = 1  ← 20,000 가스!
```

```
PUSH1 0x00   // 슬롯: 0
SLOAD        // 스택에 storage[0] 값 올림 ← 2,100 가스
```

---

## 간단한 Solidity 함수를 옵코드로

```solidity
contract Simple {
    uint256 public value;

    function setValue(uint256 _v) public {
        value = _v;
    }
}
```

`setValue(42)`를 호출할 때 EVM이 실행하는 핵심 흐름 (단순화):

```
// calldata에서 인자 읽기
PUSH1 0x04    // calldata offset (셀렉터 4바이트 뒤)
CALLDATALOAD  // 스택에 인자값(42) 올림

// 스토리지 슬롯 0에 저장
PUSH1 0x00    // 슬롯 번호
SSTORE        // storage[0] = 42
```

실제로는 더 많은 보일러플레이트 코드(함수 분기, 가스 체크 등)가 있지만, 핵심은 이렇다.

Remix에서 컨트랙트를 컴파일하고 "Compilation Details" → "OPCODES"를 보면 전체 옵코드 시퀀스를 확인할 수 있다.

---

## CALL vs DELEGATECALL vs STATICCALL

컨트랙트가 다른 컨트랙트를 호출하는 옵코드들이다. 세 가지가 있는데, 차이가 중요하다.

### CALL

일반적인 외부 호출. 대상 컨트랙트의 코드를 **대상 컨트랙트의 컨텍스트**에서 실행한다.

```
컨텍스트: msg.sender = 호출자, storage = 대상 컨트랙트 storage
```

```
A → CALL → B 실행
    B의 코드가 B의 storage를 수정
```

### DELEGATECALL

대상 컨트랙트의 코드를 **현재 컨트랙트의 컨텍스트**에서 실행한다.

```
컨텍스트: msg.sender = 원래 호출자, storage = 현재 컨트랙트 storage
```

```
A → DELEGATECALL → B의 코드 실행
    하지만 A의 storage가 수정됨!
```

이게 **프록시 패턴(Proxy Pattern)**의 기반이다. 로직 컨트랙트(B)는 교체 가능하게 두고, 프록시 컨트랙트(A)의 storage에 상태를 저장한다. 업그레이드 가능한 컨트랙트를 만드는 방법이다.

동시에 보안 취약점의 원인이 되기도 한다. 2016년 Parity 멀티시그 해킹이 DELEGATECALL 오용으로 인한 사고였다.

### STATICCALL

읽기 전용 외부 호출. 대상 컨트랙트가 상태를 변경하면(SSTORE 등) 트랜잭션이 revert된다.

Solidity의 `view` 함수를 호출할 때 내부적으로 STATICCALL이 사용된다.

## 실습: cast로 옵코드 확인

Foundry의 `cast` CLI를 쓰면 배포된 컨트랙트의 바이트코드를 바로 뜯어볼 수 있다.

```bash
# USDC 컨트랙트 바이트코드 가져오기
cast code 0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48 --rpc-url https://eth.llamarpc.com

# 옵코드로 디컴파일
cast disassemble $(cast code 0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48 --rpc-url https://eth.llamarpc.com)
```

또는 Etherscan에서 컨트랙트 주소 검색 → "Contract" 탭 → "Opcodes" 섹션에서 확인할 수도 있다.

## 가스와 옵코드의 관계

각 옵코드마다 가스 비용이 정해져 있다.

| 옵코드 | 가스 | 설명 |
|--------|------|------|
| ADD | 3 | 덧셈 |
| MUL | 5 | 곱셈 |
| SLOAD | 2,100 | 스토리지 읽기 (cold) |
| SSTORE | 20,000 | 스토리지 쓰기 (새 값) |
| CALL | 2,600+ | 외부 호출 |
| KECCAK256 | 30+ | 해시 연산 |

스토리지 읽기가 단순 연산보다 700배, 스토리지 쓰기가 7,000배 비싸다.

"반복문 안에서 상태 변수에 접근하지 말라"는 Solidity 최적화 원칙이 왜 있는지 이제 이해가 된다. 반복문이 100번 돌면 SSTORE 하나가 200만 가스가 된다.

이 가스 원리는 다음 편에서 더 깊이 다룬다.

## 용어 정리

**옵코드 (Opcode)**
EVM이 실행하는 1바이트 기계어 명령어. 256개가 정의돼 있다.

**DELEGATECALL**
대상 컨트랙트의 코드를 현재 컨트랙트의 컨텍스트(storage, msg.sender)에서 실행하는 옵코드. 프록시 패턴의 핵심이자 보안 취약점의 원인이 되기도 한다.

**STATICCALL**
상태 변경이 금지된 읽기 전용 외부 호출 옵코드. Solidity의 view/pure 함수 호출에 사용된다.
