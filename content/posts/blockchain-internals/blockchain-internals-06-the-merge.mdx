2022년 9월 15일, 이더리움이 PoW를 버리고 PoS로 전환했다. 이걸 **The Merge**라고 부른다.

결과만 보면 전기 소모가 99.95% 줄었다. 하지만 어떻게 수십억 달러 규모의 라이브 블록체인을 중단 없이 새로운 합의 알고리즘으로 교체했는지가 더 흥미롭다.

---

## 왜 "Merge"인가

The Merge 이전, 이더리움 생태계에는 두 개의 체인이 병렬로 돌아가고 있었다.

**실행 레이어 (Execution Layer)**
우리가 알던 이더리움 메인넷이다. 스마트 컨트랙트, 트랜잭션, DeFi — 이 모든 게 여기서 돌아간다. PoW로 합의를 이루고 있었다.

**합의 레이어 (Consensus Layer, 비콘 체인)**
2020년 12월에 별도로 출시된 PoS 체인이다. 이름은 **비콘 체인(Beacon Chain)**이었다. 검증자들이 ETH를 스테이킹하고 PoS 합의를 연습하는 테스트베드였다. 실제 트랜잭션이나 스마트 컨트랙트는 없었다.

The Merge는 이 두 체인을 합친 것이다. 실행 레이어의 트랜잭션 처리 능력과 합의 레이어의 PoS 합의 메커니즘이 하나로 연결됐다.

```
Before The Merge:
  실행 레이어 (PoW)   → 트랜잭션 처리 O, 합의 PoW
  합의 레이어 (PoS)   → 트랜잭션 처리 X, 합의 PoS (연습 중)

After The Merge:
  실행 레이어 + 합의 레이어가 결합
  → 트랜잭션 처리 O, 합의 PoS
```

---

## 왜 이렇게 복잡하게 했나

라이브 네트워크에서 합의 알고리즘을 교체한다는 게 얼마나 위험한 일인지 생각해보자.

비유하자면 — 시속 100km로 달리는 차의 엔진을 교체하는 것과 같다. 멈출 수가 없다. 이더리움은 한 번도 멈춘 적이 없었고, 멈출 수도 없었다.

그래서 선택한 전략이 "새 엔진을 먼저 따로 만들고 테스트해서 준비되면 연결한다"는 거였다. 비콘 체인이 1년 반 동안 독립적으로 돌아가며 검증자들이 쌓인 이유가 그거다.

---

## 두 레이어는 어떻게 통신하나: Engine API

The Merge 이후 이더리움 노드는 두 개의 소프트웨어가 함께 실행된다.

**실행 클라이언트 (Execution Client)**
- EVM 실행, 트랜잭션 처리, State Trie 관리
- 대표: Geth, Nethermind, Besu, Erigon

**합의 클라이언트 (Consensus Client)**
- PoS 합의 참여, Attestation, Proposer 선출
- 대표: Lighthouse, Prysm, Teku, Nimbus

두 클라이언트는 **Engine API**를 통해 통신한다. 이게 The Merge의 핵심 기술적 변화다.

```
합의 클라이언트           실행 클라이언트
       │                        │
       │  engine_forkchoiceUpdated  │
       │ ──────────────────────→ │  "이 블록이 헤드야"
       │                        │
       │  engine_newPayload         │
       │ ──────────────────────→ │  "이 트랜잭션들 실행해봐"
       │                        │
       │     result              │
       │ ←────────────────────── │  "유효해 / 유효하지 않아"
```

합의 클라이언트가 "다음 블록에 이 트랜잭션들 담아"라고 요청하면, 실행 클라이언트가 EVM으로 실행해서 결과를 돌려준다. 트랜잭션 실행의 결정권은 합의 클라이언트에 있고, 실제 실행은 실행 클라이언트가 한다.

## The Merge가 바꾼 것, 바꾸지 않은 것

### 바뀐 것

**합의 알고리즘**: PoW → PoS. 채굴이 사라졌다.

**에너지 소비**: 99.95% 감소. 이더리움 네트워크가 소비하는 전력이 중소 국가 수준에서 일반 기업 수준으로 줄었다.

**블록 생성 주기**: PoW에서는 평균 13초였는데, PoS에서는 정확히 12초로 고정됐다.

**블록 보상**: 채굴 보상이 없어지고, 검증자 보상만 남았다. ETH 발행량이 크게 줄었다. EIP-1559로 baseFee가 소각되는 것과 합쳐지면, 네트워크 사용량이 많을 때는 ETH가 순 감소(deflationary)하는 구조가 됐다.

### 바뀌지 않은 것

**트랜잭션 내역**: 제네시스 블록부터의 전체 역사가 그대로 유지됐다.

**스마트 컨트랙트**: 주소도, 코드도, 스토리지도 그대로다. 재배포할 필요 없었다.

**Gas 비용 구조**: The Merge 자체로 가스비가 낮아지지는 않았다. 가스비 개선은 Layer 2와 Danksharding으로 풀어야 할 문제다. "The Merge로 가스비가 낮아진다"는 당시에 퍼진 오해였다.

---

## 클라이언트 다양성이 왜 중요한가

흥미로운 사실 하나. 이더리움 재단은 특정 클라이언트 소프트웨어가 과반수를 차지하지 않도록 지속적으로 독려한다.

이유가 있다. 만약 검증자의 66% 이상이 Geth 하나만 쓰고 있다면, Geth에 버그가 생겼을 때 이더리움 전체가 Finality를 잃거나 체인이 분열될 수 있다. 실제로 과거 Geth 버그로 체인이 일시적으로 분열된 사례(2021년)가 있었다.

반면 클라이언트가 다양하게 분산돼 있으면, 한 클라이언트의 버그가 전체에 영향을 미치지 않는다. 다른 클라이언트들이 정상 동작하면서 체인을 유지한다.

이게 오픈소스 다양성이 단순한 이념이 아니라 **네트워크 안전을 위한 실질적 전략**인 이유다.

---

## 용어 정리

**실행 레이어 (Execution Layer)**
트랜잭션을 처리하고 EVM을 실행하는 레이어. Geth, Nethermind 등의 실행 클라이언트가 담당한다.

**합의 레이어 (Consensus Layer)**
PoS 합의를 담당하는 레이어. Lighthouse, Prysm 등의 합의 클라이언트가 담당한다. The Merge 이전에는 비콘 체인이라고 불렸다.

**Engine API**
실행 클라이언트와 합의 클라이언트가 통신하는 내부 인터페이스. The Merge의 핵심 기술적 연결 고리다.

**클라이언트 다양성 (Client Diversity)**
여러 팀이 만든 다양한 클라이언트 소프트웨어를 분산해서 사용하는 것. 단일 클라이언트 버그로 인한 네트워크 장애를 막는 안전망이다.
