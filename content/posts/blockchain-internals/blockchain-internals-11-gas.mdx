입문 4편에서 가스비가 뭔지는 배웠다. 트랜잭션을 보내면 수수료가 든다는 것도.

이번엔 한 단계 더 들어간다. 가스비가 실제로 어떻게 계산되는지, EIP-1559가 뭘 바꿨는지, 그리고 컨트랙트 코드에서 가스를 아끼려면 뭘 봐야 하는지.

---

## 왜 ETH를 직접 청구하지 않을까

수수료를 그냥 ETH로 청구하면 안 될까?

문제가 있다. ETH 가격은 계속 변한다. 오늘 간단한 트랜잭션이 $0.10이면 내일은 $10이 될 수도 있다. 이렇게 되면 컨트랙트를 합리적으로 설계할 수 없다.

가스는 "연산량"을 측정하는 **단위**다. 특정 연산이 얼마나 많은 리소스를 쓰는지를 가스로 표현한다. 그리고 1 가스당 ETH 환산 비용(가스 가격)은 시장에서 결정된다.

```
실제 수수료 = 소비한 가스 × 가스 가격(gwei)
```

연산량 자체는 안정적이다. `ADD` 옵코드는 항상 3 가스, `SSTORE`는 항상 20,000 가스다. ETH 가격이 올라도 연산량은 같다.

---

## 옵코드 가스 비용이 다른 이유

왜 단순 덧셈은 3 가스인데 스토리지 쓰기는 20,000 가스일까?

연산이 요구하는 실제 리소스 비용을 반영한다.

```
ADD          →  3 가스  (CPU 연산 한 번, 메모리 없음)
SLOAD        →  2,100 가스  (디스크/DB 읽기)
SSTORE       →  20,000 가스 (디스크/DB 쓰기, 모든 노드가 영구 저장)
KECCAK256    →  30 + 6*word (해시 연산량)
CREATE       →  32,000 가스 (새 컨트랙트 생성)
```

스토리지 쓰기가 비싼 이유는 명확하다. 이더리움 전체 노드가 이 값을 영구 저장해야 한다. 전 세계 수천 개 노드의 디스크에 데이터가 쓰인다.

덧셈은 CPU 연산 한 번으로 끝난다.

## EIP-1559 이전 vs 이후

### 이전 (First-Price Auction)

예전엔 가스 가격을 사용자가 직접 설정했다. 채굴자는 가스 가격이 높은 트랜잭션을 먼저 처리했다.

문제: 트래픽이 몰리면 가스 가격이 얼마여야 할지 알 수가 없었다. 경쟁적으로 높은 가격을 써내면서 실제 처리 비용보다 훨씬 비싼 수수료가 됐다.

### 이후 (EIP-1559, 런던 업그레이드)

```
수수료 = (baseFee + priorityFee) × 가스 사용량
```

**baseFee**: 프로토콜이 자동으로 설정하는 기준 수수료. 네트워크가 바쁘면 올라가고 한산하면 내려간다. 정확히 말하면, 직전 블록이 목표 가스 사용량(블록 가스 한도의 50%)보다 많으면 12.5% 올라가고, 적으면 12.5% 내려간다.

```
블록 사용량 > 50%  →  baseFee × 1.125
블록 사용량 < 50%  →  baseFee × 0.875
```

**priorityFee (tip)**: 채굴자/검증자에게 주는 팁. 빠르게 처리받고 싶으면 높게 설정한다.

**baseFee는 소각된다.** 채굴자에게 가지 않고 프로토콜이 없애버린다. 팁만 검증자 수입이다. 이것이 이더리움 공급량 감소(deflation)의 원인이다.

## gas limit vs gas used

### gas limit (가스 한도)

트랜잭션을 보낼 때 설정하는 "최대 허용 가스량"이다. 이 이상으로 가스를 쓰면 트랜잭션이 revert된다.

ETH 단순 전송: 정확히 21,000 가스. 복잡한 컨트랙트 호출: 수십만~수백만 가스.

Ethers.js가 자동으로 예상 가스를 추정해주지만(`estimateGas`), 실패 시나리오를 대비해 20~30% 여유를 두는 게 일반적이다.

### gas used (사용된 가스)

실제로 소비된 가스. gas limit을 다 쓸 필요가 없다.

```
gas limit: 100,000
gas used: 63,254

수수료 = 63,254 × (baseFee + priorityFee)
남은 가스(36,746)는 환불
```

중요한 것: **실패한 트랜잭션도 가스를 소모한다.** EVM이 실행하다가 revert를 만나면 상태는 롤백되지만, 그때까지 쓴 가스는 환불되지 않는다. 검증자가 실행 비용을 부담했기 때문이다.

---

## 블록 가스 한도

블록 하나에 담을 수 있는 총 가스량에도 상한이 있다. 현재 약 3,000만 가스.

ETH 단순 전송이 21,000 가스라면 블록 하나에 최대 약 1,400건의 단순 전송이 들어간다. 컨트랙트 상호작용이 섞이면 더 적어진다.

이게 이더리움이 초당 15~30 TPS에 묶이는 구조적 원인이다. 블록이 12초마다 생성되니, 블록당 최대 처리량이 곧 TPS 상한이 된다.

## 가스 최적화 핵심

10편에서 본 것처럼 스토리지 접근이 압도적으로 비싸다. 가스 최적화의 핵심은 SSTORE/SLOAD 횟수를 줄이는 것이다.

**패턴 1: 반복문에서 상태 변수 캐싱**

```solidity
// Bad: 반복마다 SLOAD
for (uint i = 0; i < items.length; i++) {
    total += balances[items[i]]; // SLOAD마다 2,100 가스
}

// Good: 로컬 변수에 캐싱
uint256[] memory _balances = balances; // 한 번만 SLOAD
for (uint i = 0; i < items.length; i++) {
    total += _balances[items[i]]; // MLOAD (메모리) = 훨씬 저렴
}
```

**패턴 2: storage를 0으로 만들면 환불**

스토리지 슬롯을 0으로 설정하면 가스 환불이 발생한다(EIP-3529 이후 한도 있음). 더 이상 필요 없는 데이터를 지우면 가스를 아낄 수 있다.

**패턴 3: 이벤트로 데이터 저장**

컨트랙트에서 읽을 필요 없고 외부에서 조회만 하는 데이터는 스토리지 대신 이벤트(Event)로 emit한다. 이벤트는 영수증(receipt)에 로그로 저장되며, 스토리지보다 훨씬 저렴하다.

## 용어 정리

**가스 (Gas)**
EVM 연산량을 측정하는 단위. 연산 종류마다 고정된 가스 비용이 있다.

**baseFee**
EIP-1559가 도입한 프로토콜 자동 설정 수수료. 블록 혼잡도에 따라 자동 조정되며, 소각된다.

**priorityFee (팁)**
검증자에게 주는 추가 팁. 빠른 처리를 원할 때 높게 설정한다.

**gas limit**
트랜잭션이 허용하는 최대 가스 소비량. 초과 시 revert.

**gas used**
실제 소비된 가스. gas limit 미만이면 차액은 환불된다.
