입문 시리즈에서 블록체인의 "블록"이 뭔지 배웠다. 거래 내역을 묶어서 하나의 블록으로 만들고, 그 블록들을 해시로 연결한다는 것까지.

그런데 이런 궁금증이 생길 수 있다.

> 블록 하나에 트랜잭션이 수백, 수천 개 담겨있다는데 — 내 거래가 그 블록에 실제로 있는지 어떻게 확인하지?

가장 단순한 방법은 블록 안의 트랜잭션을 처음부터 하나씩 다 뒤지는 거다. 하지만 이더리움 블록 하나에는 평균 200개 이상의 트랜잭션이 담긴다. 블록이 수천만 개 쌓인 지금, 처음부터 끝까지 다 뒤지면 현실적으로 불가능하다.

이 문제를 푸는 게 **Merkle Tree**다.

---

## 이 시리즈가 다루는 것

입문 시리즈가 "쓸 줄 아는 것"을 목표로 했다면, 이 시리즈는 "안에서 무슨 일이 벌어지는지"를 목표로 한다.

크게 다섯 묶음으로 나눠서 이야기할 건데, 첫 번째 묶음은 **데이터 구조**다. 블록체인이 데이터를 어떻게 저장하고 증명하는지 — Merkle Tree, 블록 헤더 구조, 이더리움의 상태 저장 방식 순서로 풀어낸다.

그 시작이 Merkle Tree다.

---

## 해시 복습: 지문 같은 것

Merkle Tree를 이해하려면 해시 함수를 먼저 짚고 가야 한다. 입문 1편에서 이미 다뤘지만 빠르게 복습하면:

- 어떤 데이터를 넣어도 항상 **같은 길이**의 문자열이 나온다
- 입력이 **1글자라도** 바뀌면 출력이 완전히 달라진다
- 출력값으로 입력값을 역산하는 건 **사실상 불가능**하다

```
"Alice → Bob, 1 ETH"   → a3f9c2...
"alice → Bob, 1 ETH"   → 7d1e84...  ← 대문자 하나 달라졌는데 완전히 다른 값
```

이 성질 때문에 해시는 "데이터의 지문"으로 쓰인다. 지문이 같으면 데이터가 같고, 다르면 뭔가 바뀐 것이다.

---

## 트리를 왜 만드나

트랜잭션 4개짜리 블록이 있다고 해보자.

```
Tx A: Alice → Bob, 1 ETH
Tx B: Carol → Dave, 2 ETH
Tx C: Eve → Frank, 0.5 ETH
Tx D: Grace → Henry, 3 ETH
```

이 4개를 그냥 해시 하나로 만들면 어떻게 될까.

```
hash(TxA + TxB + TxC + TxD) = X
```

"내 거래(TxA)가 이 블록에 있다"는 걸 증명하려면 TxB, TxC, TxD 전부를 검증자한테 넘겨줘야 한다. 그래야 상대방이 다시 합쳐서 X가 맞는지 확인할 수 있기 때문이다. 트랜잭션이 100만 개면 100만 개를 다 넘겨야 한다. 비효율적이다.

Merkle Tree는 이 문제를 **이진 트리 구조**로 푼다.

---

## Merkle Tree 구조

트랜잭션을 2개씩 짝지어서 해시하고, 그 결과를 또 2개씩 짝지어서 해시한다. 이걸 하나가 남을 때까지 반복한다.

```
        [Root]
       /      \
   [H-AB]   [H-CD]
   /    \   /    \
[H-A] [H-B] [H-C] [H-D]
  |     |     |     |
 TxA   TxB   TxC   TxD
```

- `H-A` = hash(TxA)
- `H-B` = hash(TxB)
- `H-AB` = hash(H-A + H-B)
- `Root` = hash(H-AB + H-CD)

맨 꼭대기에 있는 `Root`를 **Merkle Root**라고 부른다. 이 값 하나가 블록 안의 모든 트랜잭션을 대표한다.

---

## "TxA가 이 블록에 있다"를 증명하는 법

내가 TxA가 이 블록에 있다는 걸 남한테 증명하고 싶다. 상대방은 Merkle Root만 알고 있다.

이때 상대방한테 줘야 하는 건 TxA 자체와, 경로상에 있는 **형제 노드 해시**들이다.

```
필요한 것: TxA, H-B, H-CD
```

상대방이 하는 계산:

```
1. H-A = hash(TxA)            → 직접 계산
2. H-AB = hash(H-A + H-B)     → H-B를 내가 줬으니 가능
3. Root = hash(H-AB + H-CD)   → H-CD를 내가 줬으니 가능
4. 계산한 Root == 알고 있는 Root?  → 일치하면 TxA가 블록에 있는 것
```

트랜잭션이 4개일 때 필요한 데이터는 2개(H-B, H-CD). 트랜잭션이 100만 개여도 필요한 데이터는 **log₂(100만) ≈ 20개**다. 전체를 넘겨줄 필요가 없다.

---

## 이게 왜 중요하냐면

이더리움에는 세 종류의 노드가 있다.

- **풀노드**: 블록체인 전체 데이터를 갖고 있다. 모든 트랜잭션을 검증할 수 있다.
- **라이트 노드(SPV)**: 블록 헤더만 갖고 있다. 저장 공간이 적은 스마트폰 같은 환경에서 쓴다.
- **아카이브 노드**: 풀노드보다도 더 많은 역사적 상태를 저장한다.

라이트 노드는 모든 트랜잭션을 갖고 있지 않다. 그런데 Merkle Tree 덕분에 풀노드한테 소수의 해시값만 받아서도 "이 트랜잭션이 저 블록에 실제로 있는지"를 검증할 수 있다. 이 증명 방식을 **Merkle Proof** 또는 **SPV(Simplified Payment Verification)**라고 부른다.

MetaMask 같은 지갑이 내 거래 내역을 확인할 때, 풀노드 없이도 동작할 수 있는 게 이 원리 때문이다.

---

## 실제로 확인해보기

[Etherscan](https://etherscan.io)에서 아무 블록이나 들어가면 **Transactions Root** 필드가 보인다. 그게 그 블록의 Merkle Root다.

예를 들어 블록 #21,000,000의 Transactions Root가 `0xa3f...` 라면, 그 블록 안의 모든 트랜잭션을 Merkle Tree로 쌓아서 나온 루트가 저 값이다. 트랜잭션 하나라도 바뀌면 이 값이 완전히 달라진다.

---

## 용어 정리

**Merkle Root**
Merkle Tree의 맨 꼭대기 해시값. 블록 안의 모든 트랜잭션을 대표하는 단 하나의 값. 이더리움 블록 헤더에 `transactionsRoot`라는 필드로 저장된다.

**Merkle Proof**
트랜잭션 하나가 특정 블록에 포함돼 있다는 걸 증명하는 해시값들의 집합. 전체 트랜잭션 대신 log(n)개의 해시만으로 증명 가능하다.

**라이트 노드 (Light Node / SPV)**
블록 헤더만 저장하고, Merkle Proof를 통해 트랜잭션을 검증하는 노드. 풀노드에 비해 저장 공간이 극히 적다. 모바일 지갑이 이 방식을 쓴다.

---

다음 편에서는 Merkle Root가 실제로 담기는 곳인 **블록 헤더**를 해부한다. 블록 헤더 필드들을 하나씩 뜯어보면 이번 편에서 배운 Merkle Tree가 어디에 어떻게 쓰이는지 보인다.
