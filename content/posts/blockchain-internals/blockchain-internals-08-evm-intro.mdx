Part 3는 EVM이다. 스마트 컨트랙트를 배포하고 호출하는 법은 입문 시리즈에서 다뤘다. 이번에는 그 코드가 실제로 어떻게 실행되는지 — EVM 내부로 들어간다.

---

## 왜 "가상 머신"이 필요한가

이더리움 네트워크에는 수만 개의 노드가 있다. 맥북, 리눅스 서버, 윈도우 PC, 클라우드 인스턴스 — 하드웨어와 운영체제가 전부 다르다.

스마트 컨트랙트는 이 모든 노드에서 **똑같은 결과**가 나와야 한다. Alice가 Bob한테 1 ETH를 보내는 트랜잭션을 한국 노드에서 실행한 결과와 독일 노드에서 실행한 결과가 달라지면 합의가 깨진다.

이게 **결정론적 실행(Deterministic Execution)**이다. 입력이 같으면 어디서 실행해도 출력이 반드시 같아야 한다.

일반 프로그램은 이게 보장되지 않는다. OS 버전, 하드웨어, 라이브러리 버전에 따라 동작이 달라질 수 있다. 부동소수점 연산의 경우 아키텍처마다 결과가 미묘하게 다를 수도 있다.

그래서 이더리움은 **EVM(Ethereum Virtual Machine)**이라는 가상 머신을 만들었다. 어떤 컴퓨터에서 실행하더라도 동일하게 동작하도록 규격화된 실행 환경이다. Solidity 코드는 이 EVM이 이해하는 바이트코드로 컴파일돼서 모든 노드에서 동일하게 실행된다.

---

## 스택 기반 가상 머신

EVM은 **스택(Stack) 기반** 가상 머신이다.

컴퓨터 구조를 배운 적 있다면 **레지스터 기반** CPU(Intel, ARM 등)와 비교하면 이해가 쉽다.

레지스터 기반은 값을 저장하는 "서랍(레지스터)"이 여러 개 있고, 서랍끼리 연산한다.

```
// 레지스터 기반 (가상 예시)
LOAD R1, 5    // R1에 5 저장
LOAD R2, 3    // R2에 3 저장
ADD R3, R1, R2  // R3 = R1 + R2
```

스택 기반은 값을 쌓아두는 "탑(스택)"이 하나 있고, 탑의 맨 위에서 꺼내서 연산하고 다시 올린다.

```
// 스택 기반 (EVM 예시)
PUSH 5   // 스택: [5]
PUSH 3   // 스택: [5, 3]
ADD      // 3과 5를 꺼내서 더하고, 결과 8을 올림 → 스택: [8]
```

스택 기반이 구현이 더 단순하고, 검증이 쉽다. EVM 스펙을 여러 팀이 각자 구현해야 하는 상황에서(다양한 클라이언트) 이 단순함이 중요하다.

EVM의 스택은 최대 1024개까지 쌓을 수 있다. 이를 초과하면 스택 오버플로우 에러가 발생한다.

---

## EVM의 네 가지 저장 공간

EVM이 코드를 실행할 때 데이터를 저장하는 공간이 네 가지 있다. 각각 쓰임새와 비용이 다르다.

### Stack (스택)

방금 설명한 것. 연산 중에 임시로 값을 쌓아두는 공간이다. 빠르고 저렴하다. 함수 실행이 끝나면 사라진다.

최대 1024 depth, 각 슬롯은 32바이트.

### Memory (메모리)

함수 실행 중에 임시로 쓰는 공간이다. 바이트 단위로 읽고 쓸 수 있다. 스택보다 유연하지만 조금 비싸다. 마찬가지로 함수 실행이 끝나면 사라진다.

Solidity에서 `memory` 키워드를 붙이는 변수들이 여기에 저장된다.

```solidity
function process(string memory name) public { ... }
//                      ^^^^^^
//                      이 변수는 Memory에 저장
```

### Storage (스토리지)

영구 저장 공간이다. 컨트랙트의 상태 변수(state variable)들이 여기에 저장된다.

```solidity
contract Counter {
    uint256 public count; // Storage에 저장 (영구)

    function increment() public {
        uint256 temp = count + 1; // temp는 Stack에 임시 저장
        count = temp; // Storage에 쓰기
    }
}
```

영구 저장이라 읽고 쓰는 비용이 매우 비싸다. `SSTORE` 옵코드(스토리지 쓰기)는 20,000 가스가 필요한 반면, `ADD`(덧셈)는 3 가스다. 가스 최적화의 핵심이 "스토리지 접근을 최소화하는 것"인 이유가 여기 있다.

### Calldata (콜데이터)

트랜잭션이 컨트랙트를 호출할 때 넘겨주는 입력 데이터가 여기 담긴다. 읽기 전용이고 변경 불가능하다.

Solidity에서 외부 함수의 매개변수에 `calldata`를 붙이면 메모리 복사 없이 직접 읽을 수 있어서 가스를 아낄 수 있다.

## Solidity가 바이트코드가 되는 과정

```solidity
// 이 코드를
function add(uint256 a, uint256 b) public pure returns (uint256) {
    return a + b;
}
```

Solidity 컴파일러(`solc`)가 이걸 EVM 바이트코드로 변환한다.

```
60 80 60 40 52 34 80 15 61 00 0e 57 60 00 80 fd ...
```

사람이 읽을 수 없는 16진수 바이트 나열이다. 이걸 다시 사람이 읽을 수 있는 **옵코드(Opcode)**로 변환하면:

```
PUSH1 0x80
PUSH1 0x40
MSTORE
CALLVALUE
DUP1
ISZERO
...
```

바이트코드 자체가 이더리움 블록체인에 저장되고, 트랜잭션이 들어오면 EVM이 이 옵코드들을 하나씩 읽어서 실행한다.

Remix에서 컨트랙트를 컴파일하면 "Bytecode"와 "Opcodes" 탭에서 이걸 직접 확인할 수 있다.

---

## EVM은 고립된 환경이다

EVM 코드는 실행 중에 외부 세계와 직접 통신할 수 없다. 인터넷에 HTTP 요청을 날리거나, 파일을 읽거나, 현재 시각을 정확히 가져오거나 하는 게 불가능하다.

이게 결정론적 실행을 보장하는 핵심이다. 외부 I/O가 허용되면 노드마다 결과가 달라질 수 있다.

그래서 스마트 컨트랙트가 외부 데이터(예: ETH/USD 가격)가 필요하면 **오라클(Oracle)**이라는 별도 시스템을 써야 한다. 오라클이 외부 데이터를 블록체인에 올리면, 컨트랙트는 그 데이터를 읽는다. Chainlink가 대표적인 오라클 서비스다.

## 용어 정리

**EVM (Ethereum Virtual Machine)**
이더리움의 스마트 컨트랙트 실행 환경. 어떤 하드웨어·OS에서 실행해도 동일한 결과를 보장하는 스택 기반 가상 머신이다.

**결정론적 실행 (Deterministic Execution)**
같은 입력에 대해 언제 어디서 실행해도 동일한 출력이 나오는 성질. 블록체인 합의의 전제 조건이다.

**스택 기반 VM**
데이터를 스택에 쌓아두고 맨 위에서 꺼내 연산하는 방식의 가상 머신. 레지스터 기반보다 구현이 단순하다.

**Storage**
컨트랙트의 영구 저장 공간. 읽기/쓰기 비용이 매우 높다. 상태 변수가 여기 저장된다.

**Memory**
함수 실행 중에만 존재하는 임시 저장 공간. 함수가 끝나면 사라진다.

**Calldata**
트랜잭션 호출 시 전달되는 입력 데이터. 읽기 전용이며 변경 불가능하다.
