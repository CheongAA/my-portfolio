Part 4는 트랜잭션의 여정이다.

지금까지는 EVM 안에서 코드가 어떻게 실행되는지 봤다. 이번엔 시점을 더 앞으로 당겨서 — 트랜잭션이 서명된 직후, 어떻게 전 세계 노드에 도달하는지부터 시작한다.

---

## 서버가 없다

중앙 서버가 있다면 간단하다. 내 트랜잭션을 서버에 보내면, 서버가 나머지 노드들에 알린다.

이더리움에는 그런 서버가 없다. 수천 개의 노드가 직접 연결된 **P2P(Peer-to-Peer)** 네트워크다.

내가 트랜잭션을 보내면 내 지갑이 연결된 노드 하나에 도달한다. 그 노드가 이웃 노드들에 전달하고, 이웃 노드가 또 자기 이웃에 전달한다. 이 과정이 반복되면서 트랜잭션이 전체 네트워크에 퍼진다.

이걸 **가십(Gossip) 프로토콜**이라고 한다.

```
내 지갑 → 노드 A
노드 A → 노드 B, C, D
노드 B → 노드 E, F
노드 C → 노드 G, H
...
```

---

## 노드가 서로를 찾는 방법

처음 네트워크에 참여하는 노드가 다른 노드를 어떻게 발견할까?

이더리움은 **discv5(Discovery Protocol v5)**라는 P2P 노드 발견 프로토콜을 쓴다. DHT(Distributed Hash Table) 기반으로, 처음엔 하드코딩된 **부트스트랩 노드(Bootstrap Node)**에 연결해서 다른 노드 목록을 받아온다.

노드들은 이웃 노드 목록(피어 테이블)을 유지하며, 주기적으로 살아있는지 확인한다(ping/pong). 노드가 꺼지면 자연스럽게 테이블에서 빠진다.

```
신규 노드 → 부트스트랩 노드 연결
부트스트랩 → 주변 노드 목록 제공
신규 노드 → 목록의 노드들과 직접 연결
이후 → gossip으로 더 많은 노드 발견
```

노드 하나는 보통 25~50개의 이웃 피어를 유지한다.

---

## 이더리움 노드 유형

이더리움 노드가 다 같은 건 아니다.

### 풀 노드 (Full Node)

모든 블록을 다운받아 직접 검증한다. 현재 상태(state)만 저장하고, 오래된 블록 데이터는 정리할 수 있다.

대부분의 이더리움 노드가 풀 노드다. Infura, Alchemy 같은 노드 서비스도 풀 노드를 운영한다.

### 아카이브 노드 (Archive Node)

모든 블록의 모든 상태를 저장한다. "블록 1,000만 번째 시점의 내 잔액은 얼마였나" 같은 쿼리가 가능하다.

저장 공간이 수 TB에 달한다. 일반적으로 블록 탐색기(Etherscan)나 분석 서비스가 아카이브 노드를 운영한다.

### 라이트 노드 (Light Node)

블록 헤더만 다운받고, 필요할 때만 풀 노드에 데이터를 요청한다. 1편에서 배운 Merkle Proof 덕분에 트랜잭션이 블록에 포함됐는지 검증하면서도 전체 블록 데이터를 받을 필요가 없다.

저장 공간이 매우 적게 필요해 모바일 지갑, IoT 기기에 적합하다. 다만 풀 노드 지원이 필요해 독립적으로 운용하기 어렵다.

---

## 동기화 방식

새로 참여한 노드가 현재 상태를 따라잡는 방법이다.

### Full Sync

제네시스(0번 블록)부터 지금까지 모든 블록을 받아서 직접 재실행한다. 가장 안전하지만 수일이 걸릴 수 있다.

### Snap Sync

최근 상태(state snapshot)를 먼저 받고, 그 이후 블록부터 재실행한다. 현재 이더리움 표준 동기화 방식이다. 수 시간이면 완료된다.

```
Snap Sync 흐름:
1. 최근 state snapshot 다운로드 (현재 잔액, 컨트랙트 상태 등)
2. snapshot 이후 블록들 다운로드 및 재실행
3. 완료
```

## 가십 프로토콜의 효율

가십으로 트랜잭션이 퍼지는 게 효율적일까?

이론적으로 n개의 노드가 있을 때, 각 노드가 k개의 이웃에게 전달하면 log_k(n)번의 라운드로 전체에 퍼진다.

이더리움 메인넷에 약 10,000개의 풀 노드가 있다고 가정하면, 각 노드가 25개 이웃에게 전달할 때:

```
log_25(10,000) ≈ 2.9 라운드
```

이론상 3번의 전달 라운드로 전체 네트워크에 퍼진다. 실제로 이더리움 트랜잭션은 몇 초 내에 대부분의 노드에 도달한다.

단, 중복 전달이 많다. 이미 받은 트랜잭션은 다시 전달하지 않도록 각 노드가 트랜잭션 해시를 캐싱한다.

## 실행 레이어 vs 합의 레이어의 네트워크

The Merge 이후 이더리움은 두 레이어로 나뉜다(6편 참고). P2P 네트워크도 두 개다.

- **실행 레이어 P2P**: 트랜잭션 전파, devp2p 프로토콜
- **합의 레이어 P2P**: 블록, 투표 메시지 전파, libp2p 프로토콜

트랜잭션은 실행 레이어 P2P로 퍼진다. 블록 제안은 합의 레이어 P2P로 퍼진다.

## 용어 정리

**P2P 네트워크**
중앙 서버 없이 노드들이 직접 연결되는 네트워크 구조.

**가십 프로토콜 (Gossip Protocol)**
각 노드가 받은 정보를 이웃 노드들에 전달하는 방식. 소문이 퍼지는 것처럼 전체 네트워크에 전파된다.

**discv5**
이더리움의 P2P 노드 발견 프로토콜. DHT 기반으로 동작한다.

**부트스트랩 노드**
신규 노드가 처음 연결하는 하드코딩된 진입점 노드. 다른 피어 목록을 제공한다.

**Snap Sync**
최근 state snapshot을 먼저 다운받아 동기화 시간을 단축하는 방식. 현재 이더리움 표준 동기화 방법.
