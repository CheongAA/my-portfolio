지난 편에서 개인키와 지갑을 다뤘다. MetaMask를 설치했다면 이제 이더리움이 뭔지 알 차례다.

비트코인이랑 이더리움, 둘 다 블록체인이라고 불린다. 같은 기반 기술인데 왜 따로 존재하는 걸까? 단순히 "더 발전된 버전"이 아니다. 처음부터 목적이 달랐다. 그 차이가 생각보다 훨씬 크다.

## 비트코인: 돈 보내는 기계

비트코인은 2008년 사토시 나카모토가 발표한 9페이지짜리 논문에서 시작됐다. 제목이 "Bitcoin: A Peer-to-Peer Electronic Cash System"이다. 목표가 논문 제목에 다 들어있다. **중앙 기관 없이 A에서 B로 돈을 보내는 것.**

2008년은 금융위기가 터진 해다. 은행이 탐욕적으로 운영되고, 정부가 세금으로 이를 구제하는 과정을 보면서 "은행 없이 신뢰할 수 있는 돈 전송 시스템"에 대한 수요가 생겼다. 비트코인은 그 대답이었다.

비트코인은 이 목적에 집중해서 설계됐다. 비트코인 스크립트라는 언어가 있는데, 의도적으로 단순하게 만들어졌다. "이 주소가 올바른 서명을 제공하면 코인을 보낸다" 수준의 조건만 지원한다. 루프도 없고, 복잡한 로직도 없다.

이게 단점처럼 보이지만 의도된 설계다. 복잡한 기능이 없으면 공격 표면이 줄어든다. 돈을 다루는 시스템에서 예측 불가능한 동작은 재앙이다. 비트코인은 "단순하고 안전하게"를 선택했다.

### UTXO 모델: 계좌가 아니라 잔돈 개념

비트코인을 이해하려면 UTXO를 알아야 한다. 은행 계좌처럼 "내 잔액이 1 BTC"라는 식의 계좌 개념이 없다.

대신 **UTXO(Unspent Transaction Output, 미사용 트랜잭션 출력)**라는 모델을 쓴다.

지폐로 비유하면 쉽다.

지갑에 5만원짜리 지폐 하나와 1만원짜리 지폐 두 장이 있다고 하자. 합계는 7만원이지만, 실제로는 세 장의 지폐가 있는 것이다. 2만원짜리 물건을 살 때 5만원짜리를 내면 3만원 거스름돈을 받는다. 7만원이라는 "잔액"을 직접 쪼개는 게 아니라, 지폐를 쓰고 거스름돈을 받는 방식이다.

비트코인이 그렇다.

```
[트랜잭션 흐름 예시]

누군가에게 0.5 BTC를 받음  →  UTXO #1: 0.5 BTC (내 주소)
누군가에게 0.3 BTC를 받음  →  UTXO #2: 0.3 BTC (내 주소)

"내 잔액" = 0.5 + 0.3 = 0.8 BTC  (두 UTXO의 합)

0.2 BTC를 보내고 싶다면?
  UTXO #1 (0.5 BTC)을 INPUT으로 사용
  OUTPUT #1: 0.2 BTC → 상대방 주소
  OUTPUT #2: 0.3 BTC → 내 주소 (거스름돈)
  (수수료 0.0xxx BTC는 채굴자가 가져감)
```

UTXO는 한 번 쓰면 사라진다. 새 트랜잭션의 OUTPUT이 새 UTXO가 된다. 비트코인 네트워크는 어떤 UTXO가 아직 사용되지 않았는지를 추적한다.

이 모델이 병렬 처리에 유리하고, 이중 지불을 막는 데 효과적이다. 단, 스마트 컨트랙트처럼 복잡한 상태 관리에는 맞지 않는다.

## 이더리움: 프로그래밍 가능한 블록체인

이더리움은 2013년 비탈릭 부테린이 19살에 설계했다. 그가 던진 질문은 이것이었다.

> "비트코인이 돈을 보내는 규칙을 코드로 실행할 수 있다면, 더 복잡한 규칙도 코드로 실행할 수 있지 않을까?"

비트코인 커뮤니티에 이 아이디어를 제안했지만 받아들여지지 않았다. 그래서 직접 만들었다. 핵심 차이는 하나다. **이더리움의 블록에는 코드가 담긴다.**

단순 송금을 넘어 "만약 A가 이 조건을 충족하면, 자동으로 B에게 자산이 이전된다" 같은 복잡한 로직을 블록체인 위에서 실행할 수 있다. 이 코드를 **스마트 컨트랙트**라고 부른다.

---

## EVM: 탈중앙화된 컴퓨터

코드를 실행하려면 컴퓨터가 필요하다. 이더리움은 **EVM(Ethereum Virtual Machine)**이라는 가상 컴퓨터를 정의했다.

EVM은 특정 하드웨어나 OS에 종속되지 않는 추상화된 가상 머신이다. 이더리움 노드를 실행하는 컴퓨터는 각자 자신의 EVM을 실행하고, 같은 컨트랙트 코드를 같은 방식으로 처리한다. 전 세계 수만 개의 노드가 동일한 코드를 동일하게 실행한다.

이게 왜 중요한가?

일반 웹 서비스는 AWS나 GCP 같은 서버 위에서 돌아간다. 서비스 회사가 마음을 바꾸면 서버를 끄거나, 특정 사용자를 차단하거나, 코드를 바꿀 수 있다. 혹은 해킹이나 파산으로 서버가 내려갈 수도 있다.

EVM에 올라간 코드는 그렇지 않다. 코드가 전 세계 노드에 복사돼 있기 때문에, 일부 노드가 꺼져도 나머지가 계속 실행한다. 배포한 개발자도 코드를 바꿀 수 없다. 어떤 정부도 특정 스마트 컨트랙트를 강제로 중단시킬 수 없다.

이더리움을 "**월드 컴퓨터**"라고 부르는 이유가 여기 있다.

**EVM 바이트코드**

Solidity로 작성한 코드는 컴파일되면 EVM 바이트코드로 변환된다. 이 바이트코드가 실제로 블록체인에 저장되고 실행된다.

```
Solidity 코드 → 컴파일러(solc) → EVM 바이트코드 → 블록체인에 저장
```

EVM은 스택 기반 가상 머신으로, 약 140개의 opcode(연산 명령)를 가진다. `ADD`, `MUL`, `SSTORE`(스토리지에 저장), `CALL`(다른 컨트랙트 호출) 같은 것들이다. 각 opcode마다 소비하는 gas 비용이 정해져 있다.

## 계정 모델: UTXO 대신 잔액을 직접 기록

이더리움은 비트코인의 UTXO 모델 대신 **계정(Account) 모델**을 채택했다. 은행 계좌처럼 각 주소에 잔액이 직접 기록된다.

```
[이더리움 상태 테이블 (단순화)]

주소                                    잔액(Wei)     nonce
0xAlice...  →  1,000,000,000,000,000,000  (1 ETH)   5
0xBob...    →    500,000,000,000,000,000  (0.5 ETH)  2
0xContract... →  0                                   42
```

Alice가 Bob에게 0.1 ETH를 보내면, Alice 잔액이 줄고 Bob 잔액이 늘어난다. 단순하고 직관적이다.

**nonce**는 각 주소에서 보낸 트랜잭션 수다. 트랜잭션마다 1씩 증가한다. 같은 nonce로 두 번 트랜잭션을 보낼 수 없기 때문에 재전송 공격(replay attack)을 막는다.

**이더리움 계정의 두 종류**

이더리움 계정은 두 가지 타입이 있다.

**1. EOA (Externally Owned Account)**

개인키로 제어하는 일반 지갑이다. MetaMask에서 만든 주소가 이것이다.

- 개인키가 있으면 트랜잭션을 서명해서 보낼 수 있다
- 코드가 없다
- 잔액(ETH)을 보유할 수 있다

**2. Contract Account**

스마트 컨트랙트를 배포하면 생기는 계정이다.

- 코드가 있다 (EVM 바이트코드)
- 개인키가 없다. 외부에서 트랜잭션을 보내야만 실행된다
- 잔액(ETH)을 보유할 수 있다
- 자체적인 스토리지를 가진다 (상태 변수가 여기 저장됨)

이 두 타입이 서로 트랜잭션을 주고받으며 이더리움 생태계가 돌아간다.

## 가스비(Gas): 계산에 내는 수수료

이더리움에서 뭔가를 실행하려면 **가스비**를 낸다. 단순 ETH 전송도, 스마트 컨트랙트 호출도 모두 가스비가 든다.

**왜 필요한가**

EVM은 **튜링 완전(Turing-complete)**하다. 어떤 계산도 이론적으로 실행할 수 있다는 뜻이다. 무한 루프도 가능하다.

```solidity
function infiniteLoop() public {
    while (true) {} // 이 코드를 실행하면?
}
```

만약 가스비가 없다면 악의적인 사람이 이런 코드를 배포해서 네트워크 전체를 마비시킬 수 있다. 가스비가 방지책이다. EVM은 각 연산마다 가스를 소비하고, 가스가 떨어지면 실행이 중단된다. 이미 소비된 가스는 돌려받지 못한다.

**가스비 계산**

EIP-1559(2021년 런던 하드포크) 이후 가스비 구조:

```
최종 가스비 = Gas Used × (Base Fee + Priority Fee)
```

- **Gas Used**: 트랜잭션이 실제로 소비한 연산량
- **Base Fee**: 네트워크 혼잡도에 따라 프로토콜이 자동 결정. 이 금액은 소각(burn)된다 — ETH 공급량이 줄어드는 효과
- **Priority Fee (팁)**: 검증자에게 주는 팁. 높을수록 트랜잭션이 빨리 처리된다

MetaMask에서 트랜잭션 보낼 때 나오는 "느림 / 보통 / 빠름" 옵션이 Priority Fee를 다르게 설정하는 것이다.

**Gas Limit 설정**

트랜잭션을 보낼 때 **Gas Limit**도 설정한다. "이 트랜잭션에 최대 이만큼의 가스만 쓴다"는 뜻이다. 실행 중 Gas Limit을 초과하면 트랜잭션이 실패하고 소비된 가스는 환불되지 않는다. 미사용 가스는 돌려받는다.

```
Gas Limit: 100,000  (내가 설정한 최대치)
Gas Used:   65,000  (실제 소비)
환불 가스:  35,000  (돌려받음)
```

**작업별 가스 소비량**

EVM opcode마다 소비하는 가스가 다르다. 스토리지에 값을 쓰는 것(SSTORE)이 가장 비싸고, 단순 계산은 상대적으로 싸다.

| 작업 | Gas 소비량 (대략) |
|------|------|
| ETH 전송 | 21,000 |
| ERC-20 토큰 전송 | 45,000~65,000 |
| Uniswap 토큰 스왑 | 150,000~200,000 |
| NFT 민팅 | 50,000~300,000 |
| 스마트 컨트랙트 배포 | 500,000~2,000,000+ |

가스비 최적화가 스마트 컨트랙트 개발의 중요한 주제인 이유다. 같은 기능이라도 코드를 어떻게 짜느냐에 따라 가스비가 크게 달라진다.

## 단위: Wei, Gwei, Ether

이더리움 코인 단위가 세 개다.

| 단위 | 크기 | 주로 쓰이는 곳 |
|------|------|------|
| Wei | 1 (가장 작음) | 컨트랙트 코드 내부 계산 |
| Gwei | 1,000,000,000 Wei | 가스비 표시 |
| Ether (ETH) | 1,000,000,000 Gwei (= 10¹⁸ Wei) | 실제 거래, 가격 표시 |

**왜 Wei를 기본 단위로 쓰나?**

부동소수점 문제 때문이다. 컴퓨터에서 `0.1 + 0.2 = 0.30000000000000004`가 되는 그 문제다. 돈을 다루는 코드에서 부동소수점 오류는 치명적이다.

Wei를 최소 단위로 쓰면 모든 금액을 정수로 표현할 수 있다. 정수 연산은 오차가 없다. 그래서 스마트 컨트랙트 내부는 항상 Wei 단위로 계산한다.

1 ETH = 1,000,000,000,000,000,000 Wei (10의 18승).

Ethers.js에서는 변환 유틸리티를 제공한다.

```javascript
import { parseEther, formatEther, parseUnits, formatUnits } from 'ethers'

// ETH → Wei 변환 (사용자 입력을 컨트랙트에 넘길 때)
parseEther("1.5")
// → 1500000000000000000n

// Wei → ETH 변환 (컨트랙트 결과를 화면에 보여줄 때)
formatEther(1500000000000000000n)
// → "1.5"

// Gwei ↔ Wei (가스비 계산)
parseUnits("30", "gwei")          // → 30000000000n
formatUnits(30000000000n, "gwei") // → "30.0"
```

## The Merge: PoW → PoS 전환

2022년 9월 이더리움은 "The Merge"라고 불리는 대규모 업그레이드를 단행했다. 합의 알고리즘을 PoW(작업증명)에서 PoS(지분증명)으로 전환했다.

2편에서 다뤘듯 PoW는 채굴(채굴자가 수학 문제를 풀어 블록 생성)을 통해 합의하는 방식이다. 전기 소비가 엄청나다. 이더리움은 비트코인 다음으로 큰 PoW 네트워크였다.

The Merge 이후:
- **전력 소비 99.95% 감소** — 중소 국가 수준의 전력 소비가 스위스 한 도시 수준으로 줄었다
- **ETH 발행량 감소** — 채굴자에게 지급되던 블록 보상이 대폭 줄었다. EIP-1559 소각과 합쳐지면서 ETH가 디플레이션 자산이 되는 시기도 있었다
- **보안 유지** — 공격하려면 전체 스테이킹된 ETH의 33% 이상을 보유해야 한다

비트코인은 여전히 PoW를 쓴다. 이 부분도 두 체인의 차이다.

## EVM 호환 체인들

EVM이 오픈소스로 공개돼 있기 때문에, 같은 EVM을 탑재한 블록체인이 많다. 이더리움에서 작성한 Solidity 코드가 거의 그대로 실행된다.

| 체인 | 특징 |
|------|------|
| Polygon | 이더리움 사이드체인. 빠르고 저렴 |
| Arbitrum | 이더리움 Layer 2. 이더리움 보안 그대로 활용 |
| Optimism | 이더리움 Layer 2. OP Stack 기반 |
| BNB Chain | 바이낸스 운영. 중앙화 요소 있음 |
| Avalanche C-Chain | EVM 호환, 자체 합의 알고리즘 |

이 체인들을 **EVM 호환 체인**이라고 한다. Solidity 컨트랙트를 그대로 배포할 수 있고, Ethers.js로 연결하는 코드도 네트워크 설정만 바꾸면 동일하게 작동한다. 이더리움 개발 스킬이 이 체인들에도 그대로 통한다.

| 스크립팅 | 단순 (비튜링 완전) | EVM (튜링 완전) |
| 스마트 컨트랙트 | 불가 | 가능 |
| 합의 알고리즘 | PoW | PoS (2022 전환) |
| 최소 단위 | Satoshi (10⁻⁸ BTC) | Wei (10⁻¹⁸ ETH) |
| 수수료 단위 표시 | sat/vB | Gwei |

비트코인이 "디지털 금"이라면, 이더리움은 "프로그래밍 가능한 블록체인"이다. 어느 쪽이 낫다는 비교는 의미가 없다. 목적 자체가 다르다.

이 시리즈에서는 스마트 컨트랙트와 dApp 개발이 목적이기 때문에 이후 내용은 전부 이더리움(과 EVM 호환 체인) 기반이다.

다음 편에서는 EVM 위에서 실행되는 스마트 컨트랙트가 정확히 뭔지, 왜 한 번 배포하면 수정이 안 되는지를 다룬다.
