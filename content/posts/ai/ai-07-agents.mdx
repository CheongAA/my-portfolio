지금까지 다룬 MCP와 Function Calling은 AI에게 도구를 붙여주는 방법이었다. AI 에이전트는 그 위에 하나가 더 얹힌다. **루프**다.

---

## 에이전트와 일반 AI의 차이

일반 AI 응답:

```
나: PR 설명 작성해줘
AI: [PR 설명 텍스트 반환]
끝.
```

AI 에이전트:

```
나: 이 레포의 최근 커밋을 보고 PR 설명 작성해줘

에이전트:
  1. git log 도구로 최근 커밋 조회
  2. 변경된 파일 목록 조회
  3. 주요 변경사항 분석
  4. PR 설명 작성
  5. 초안이 불완전하면 추가 정보 더 조회
  6. 최종 PR 설명 반환
```

에이전트는 목표를 완료하기 위해 **여러 단계를 스스로 계획하고 실행한다.** 필요한 만큼 도구를 쓰고 판단을 반복하다 목표가 완료되면 멈춘다.

---

## ReAct 루프 — 내부에서 벌어지는 일

에이전트가 작동하는 구조다. Reason + Act의 합성어.

```
[생각] 먼저 최근 커밋을 확인해야겠다.
[행동] git_log(limit=10) 호출
[관찰] "feat: add user auth, fix: login redirect, refactor: db connection..."

[생각] auth 관련 변경이 주요 내용이다. 파일을 더 봐야겠다.
[행동] git_diff("HEAD~3", "HEAD") 호출
[관찰] [diff 내용]

[생각] 이제 PR 설명을 쓸 수 있다.
[행동] 최종 결과 반환
```

Think → Act → Observe가 목표 완료까지 반복된다. 중간에 새로운 정보가 나오면 계획을 수정한다.

## 실제로 어떻게 쓰는가

### Claude Code에서 에이전트 쓰기

Claude Code는 기본적으로 에이전트 모드로 동작한다. 별도로 켤 필요 없이 그냥 쓰면 된다.

```bash
# 터미널에서 실행
claude

# 또는 특정 디렉터리에서
claude --dir /path/to/project
```

실행하면 대화창이 열린다. 복잡한 작업을 그냥 말하면 알아서 여러 단계로 처리한다.

```
나: 이 프로젝트에서 사용하지 않는 import를 전부 찾아서 제거해줘.

Claude Code:
  [src 디렉터리 탐색]
  [각 파일의 import 분석]
  [실제로 사용되지 않는 import 목록 작성]
  [파일들 수정]
  → 완료. 12개 파일에서 23개 미사용 import 제거됨.
```

한 번 지시하면 Claude Code가 파일을 직접 읽고, 수정하고, 완료까지 처리한다. 각 단계에서 뭘 하는지 실시간으로 보여준다.

**실제 사용 예시:**

```
# 버그 수정
"로그인 후 리다이렉트가 안 되는 버그 고쳐줘."

# 기능 구현
"다크모드 토글 기능 추가해줘. 설정은 localStorage에 저장."

# 리팩터링
"src/utils에 있는 함수들 중 중복되는 거 합쳐줘."

# 분석
"이 코드베이스에서 성능 병목이 될 수 있는 부분 찾아줘."
```

각 작업마다 Claude Code가 필요한 파일을 열고, 분석하고, 코드를 수정하고, 결과를 보여준다.

### Cursor에서 에이전트 쓰기

Cursor는 에이전트 모드를 명시적으로 선택해야 한다.

1. 우측 채팅 패널 열기 (`Cmd + L`)
2. 입력창 왼쪽 드롭다운에서 **Agent** 선택 (기본값은 Chat)
3. 요청 입력

```
Agent 모드:
"이 파일에서 에러 처리가 없는 async 함수들 찾아서 try-catch 추가해줘."

→ Cursor가 파일을 탐색하고, 해당 함수들을 찾고, 직접 수정함
```

**Chat 모드 vs Agent 모드 차이:**

|             | Chat 모드                      | Agent 모드                  |
| ----------- | ------------------------------ | --------------------------- |
| 파일 수정   | 코드 제안만 (직접 적용해야 함) | 직접 수정                   |
| 여러 파일   | 하나씩                         | 여러 파일 동시              |
| 도구 사용   | 제한적                         | 파일 읽기/쓰기, 터미널 실행 |
| 적합한 용도 | 빠른 질문, 코드 설명           | 복잡한 작업, 구현           |

---

### ChatGPT에서 에이전트처럼 쓰기

ChatGPT는 공식적인 "에이전트 모드"는 없지만, **GPT-4o with tools**를 쓰면 비슷하게 동작한다.

- **웹 검색 켜기**: 최신 정보가 필요한 작업
- **Code Interpreter**: 코드 실행, 파일 분석, 데이터 처리
- **파일 업로드**: 특정 문서나 코드베이스 기반 작업

```
[Code Interpreter 켜고]
"이 CSV 파일 데이터를 분석해서 이상치를 찾아줘."

→ ChatGPT가 Python 코드를 작성하고, 실행하고, 결과를 그래프로 보여줌
```

ChatGPT에서 에이전트 수준의 작업이 필요하면 OpenAI의 **Responses API**나 **Assistants API**를 코드로 직접 구성하는 방법도 있다.

## 에이전트에게 효과적으로 지시하는 법

에이전트는 강력하지만 잘못 지시하면 엉뚱한 방향으로 간다.

**구체적인 목표와 범위를 함께:**

```
❌ "코드 리팩터링해줘."

✅ "src/api/ 디렉터리의 함수들에서 중복 에러 처리 로직을 공통 함수로 분리해줘.
    테스트 파일은 건드리지 말고, 기존 동작은 유지해."
```

**완료 조건을 명시:**

```
✅ "모든 API 엔드포인트에 rate limiting 추가해줘.
    완료되면 어떤 파일을 수정했는지 목록 보여줘."
```

**중간 확인이 필요하면 요청:**

```
✅ "각 파일 수정 전에 뭘 할 건지 먼저 알려줘. 내가 확인하고 나서 진행해."
```

## 에이전트가 실패하는 패턴

강력하지만 이런 경우에 잘 실패한다.

**무한 루프:** 목표가 불명확하면 같은 작업을 반복한다. "코드를 더 좋게 만들어줘"는 끝이 없다.

**맥락 소실:** 루프가 길어지면 초기 목표를 잃는 경우가 생긴다. 중간 결과에 집착하다 방향이 틀어진다.

**과도한 실행:** 허락 없이 파일을 삭제하거나 외부 API를 호출하는 경우. 범위를 처음부터 명확히 줘야 한다.

**해결책:** 작업을 작게 나눠서 단계별로 확인하면서 진행하는 게 안전하다. 한 번에 큰 작업을 맡기는 것보다 나눠서 맡기는 게 결과가 낫다.

---

## 언제 에이전트가 필요한가

모든 작업에 에이전트가 필요하진 않다.

**에이전트가 유리한 경우:**

- 여러 파일을 탐색하고 수정해야 할 때
- 각 단계의 결과를 보고 다음 단계를 결정해야 할 때
- 반복적이고 시간이 걸리는 작업을 자동화할 때

**단순 대화로 충분한 경우:**

- 한 번 물어보면 답이 나오는 것
- 코드 설명이나 리뷰
- 빠른 피드백이 필요할 때

단순한 작업에 에이전트를 쓰면 오히려 느리다. 도구 목록 보여달라는 건 그냥 물어보면 된다.
